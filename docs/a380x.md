### FlyByWire A380X Specific Patterns

`FlyByWireA380Definition` (code `FBW_A380`), branch `fly-by-wire-A380-integration`.

**MSFS-2024-native modular rebuild (FBW #10758, 2026-06) — MSFSBA-impact audit.** The A380X was repackaged as a native MSFS 2024 modular aircraft (new SimObject layout, lighting/perf work, a cabinless "No Cabin" variant). **It is NOT a systems/variable overhaul — verified MSFSBA-safe:** (1) **detection unchanged** — both variants keep `icao_type_designator A388` / `atc_model A380` (the def is menu-selected, not title-detected, so the new marketing Title "FlyByWire A380X (A380-842)" is irrelevant; GSX/docking `A388` resolution intact); (2) **the whole Coherent stack still resolves** — every view title-needle exists unchanged (`A380X_MFD`/`EWD`/`SDv2`/`ND`/`PFD`/`FCU`/`RMP`/`SYSTEMSHOST`, `- EFB`); (3) **no renamed/removed L:vars** — the systems-variable docs (`a380-simvars.md`/`a380x-input-events.md`/`a380x-private-local-vars.md`) were last touched 2026-05-19 by the fuel-system #10641, NOT the rebuild. **New FWS abnormal non-sensed checklists (#10253: RELIGHT IN FLIGHT, TAILPIPE FIRE, VOLCANIC ASH ENCOUNTER, BOMB ON BOARD, SEVERE TURBULENCE)** render through the same FwsCore/`.EclLine` path, so the existing EWD/ECL scrape reads them automatically (no code). **Behaviour XML PATH MOVED:** `FlyByWire_A380_842\model\behaviour\` → `FlyByWire_A380X\attachments\flybywire\Part_Interior_Cockpit\model\behaviour\` (tool-doc references updated). **The only genuinely-new pilot vars** were 12 cockpit-furniture bool toggles in the new `interactive-parts.xml` that MSFSBA lacked — added to the Cockpit panel as `Sel` combos (meal tables, keyboard trays, CAS L/R + OIT + physical cockpit-door-open panels, and the 4 armrest STOW toggles); same `(L:VAR) ! (>L:VAR)` family as the existing table/footrest toggles, routed through the prefix-less calc-path allowlist in `HandleUIVariableSet`. Branch `feat/a380-native-2024-cockpit` (off main). **Stow polarity (0/1) on the armrest STOW combos is not visually confirmed — flip the value dict if a control reads inverted in-sim.**

**No-injection MCDU/flyPad transport.** The A380X MFD/MCDU and flyPad EFB are read AND driven live through the MSFS Coherent GT remote debugger (`http://127.0.0.1:19999`, WebKit inspector, `Runtime.evaluate`) — NOT via a Community-folder mod or HTTP bridge. `SimConnect/CoherentDebuggerClient.cs` (MFD, resolves the view by title needle `A380X_MFD`) and `SimConnect/CoherentEFBClient.cs` (flyPad, title needle `- EFB`) both implement `IMcduBridge`, so `FBWA380MCDUForm`/`FBWA380EFBForm` are transport-agnostic. The in-page agents are `Resources/coherent-a380-agent.js` and `coherent-flypad-agent.js` — **ES5 only** (Coherent GT = Chromium 49: `var`, no arrow funcs, no `String.includes`, top-level try/catch). MFD page-ids shift each session → always resolve by TITLE. **No Community-folder injection, no HTML patching, no sim restart** — the debugger is the sole A380 transport (all injection code was removed; PMDG still uses its own `EFBBridgeServer`). **Developer Mode is NOT required** — verified empirically on MSFS 2024 (cold boot, Dev Mode off): port 19999 is opened by the sim itself and `/pagelist.json` + the WebSocket `Runtime.evaluate` inspector protocol both work, so the debugger transport runs with no Dev Mode and no dev toolbar.

**flyPad = WebView2 "browser mode".** `FBWA380EFBForm` renders the scraped flyPad elements as a real semantic HTML document inside an embedded **WebView2** (already a dep; PMDG EFB precedent), so the screen reader gets native browse mode (H headings, K links, arrow-by-line, real `<input>`s, `aria-live` page-change). NO http server — `NavigateToString(PageHtml)` once, then `ExecuteScriptAsync("window.__render(<json>)")` per update; the page posts click/set/refresh back via `chrome.webview.postMessage` → `WebMessageReceived` → `EnqueueCommand`. The native flat-list is a SILENT auto-fallback only if WebView2 init fails (no user toggle). **Click index gotcha:** the agent's `clickElement`/`setValue` look up the STAMPED `data-fbwa380-efb-idx`, which diverges from the list position (the list is sorted+deduped after stamping) — so `CoherentEFBClient` carries it as `items.N.aidx` and the form sends THAT, not the loop index. **flyPad div-buttons:** many action buttons (e.g. modal Cancel/Confirm) are styled `<div>`s with no `button`/`btn` token or role — the agent classifies a `hover:`-styled leaf with its own short text as a button (interactive flyPad elements carry Tailwind `hover:` classes; static text never does). Value inputs (PAX/KGS) have no label metadata → `labelFor` reads the field's parent unit text before falling back to the section heading.

**ND OANS / BTV (Ctrl+Shift+B) — DATA-ONLY, ZERO-RENDER (2026-06, latest; supersedes all earlier scrape/TabControl OANS notes).** `Forms/FBWA380/FBWA380OansForm.cs` is a native WinForms `TabControl` (**Map and BTV / Airport / Status**) driven by `SimConnect/CoherentNDClient.cs` + `Resources/coherent-oans-agent.js` — but the agent is now **data-only**: it READS OANS/BTV state from the ND JS instance objects (`document.querySelector('a380x-nd').fsInstrument.oansRef.instance` → `btvUtils` / `labelManager.labels` / `fmsDataStore` / `dataAirportIcao` / `dataAirportName`) plus OANS L:vars, and WRITES via `btvUtils` methods + the msfs-sdk EventBus (`bus.pub`). **It NEVER writes `A32NX_EFIS_L_ND_RANGE`, forces panel `.style.visibility`, or scrapes the control-panel DOM.** ⚠️ The earlier `ensureAirportZoom`+DOM-scrape design forced a continuous full-airport map render on every poll and exhausted host commit memory (paging-file-too-small errors that froze the whole machine) — **do NOT reintroduce any forced render/zoom/visibility or DOM `scrape`/`enumerate`.** Agent API: `ping()`, `snapshot()` → `{ok,available,failed,airport,fms,btv,manual,airac}`, `armRunway(name)`, `armExit(name)`, `clearBtv()`, `displayAirport(icao)` (one-shot AMDB load — USER ACTION ONLY, never on a poll), `setManualStopDistance(m)`. The client polls `snapshot()` read-only (1 s), raises a dedup'd **`oans_state`** push (+ **`oans_connected`**), and exposes commands `get_snapshot` / `oans_arm_runway` / `oans_arm_exit` / `oans_clear` / `oans_display_airport` / `oans_set_manual_stop`. **Two tiers:** with Navigraph maps loaded (`A32NX_OANS_AVAILABLE=1`) the runway/exit pick-lists + arming + predicted dry/wet/live stop, ROT, turnaround and the runway-ahead QFU all work; without Navigraph the form shows a **Manual BTV** section — runway length (ARINC `A32NX_OANS_RWY_LENGTH`), type a stop distance → `oansManualStoppingDistance` bus pub, then "select the landing runway in the FMS". The top banner states which tier is active. **Visual-only OANS controls are intentionally REMOVED** (the RWY/TWY/STAND/OTHER entity picker, ADD CROSS/FLAG, CENTER MAP, the permanently-disabled LDG SHIFT) — they have no audio meaning for a blind pilot (STAND = parking stands, OTHER = de-icing areas — both map-locate-only). The runway-ahead advisory naming (`btvUtils.rwyAheadQfu`) shows in the window readout; an **always-on** named advisory (window closed) is deferred (it would need a persistent ND read socket — the generic SimVar `A32NX_OANS_WORD_1` bit-11 advisory in the def stays the always-on path). **Navigraph is user-side:** the AMDB airport maps are fetched from the online `https://amdb.api.navigraph.com` API using the user's Navigraph login in the flyPad EFB (Shift+T) — there is NO local file to fetch and MSFSBA must never try to; the aircraft loads the maps and MSFSBA reads what's loaded. **Exit pick-list is FILTERED to valid exits for the armed runway** — `snapshot().btv.exits` returns `agent.validExitsForArmed()`, a READ-ONLY (pure airport-local-metre math, NO canvas draw, NO mutation) verbatim port of FBW `OansBrakeToVacateSelection.selectExitFromOans`'s accept/reject test (400 m touchdown-zone, 50 m centerline, 120° gates; falls back to the full list only if the armed-runway threshold refs aren't set). Without it the combo listed all airport exits alphabetically and the first (e.g. OMDB K-series for rwy 12L) were silently rejected. **Multiple exit-line features can share ONE name (live-verified KSFO 28R / exit "Q", 2026-06): a taxiway often exposes a far-off-runway segment AND a valid on-runway one under the same text.** `validExitsForArmed` whitelists a NAME when ANY of its features passes (and sets the `seen` dedup only on ACCEPT, so an invalid first feature can't block a valid later one), so the list correctly shows "Q". But `armExit` used to pick the FIRST feature by name (`findLabel`) — at KSFO that was the off-runway "Q" (230 m off the 28R centerline) → `selectExitFromOans` rejected it → "Exit Q not valid … (wrong side or too close to threshold)" even though Q *was* valid. **FIX:** `armExit(name)` now iterates EVERY same-named exit-line feature and arms the first one `selectExitFromOans` accepts (mirroring a sighted pilot clicking the correct map label). **The per-feature `selectExitFromOans` call is wrapped in its OWN `try/continue` (not one try/catch around the whole loop)** — a throw on an early/malformed same-named feature must fall through to the next candidate, not abort the loop and reject a valid later feature; the last error is reported only if NO feature armed. Likewise `armRunway`'s threshold-search loop guards `feats[k] && feats[k].properties` (it runs outside the method's try, so an unguarded `.properties` deref on a partial AMDB feature would throw uncaught out of `armRunway`). This is SAFE re the memory rule below: it loops only over the 1–2 features sharing the SELECTED name (not all ~290 exits), and a REJECTED feature returns from `selectExitFromOans` BEFORE `drawBtvLayer` — so at most ONE canvas redraw (the accepted feature) occurs, same as before. **Exit list is sorted closest-first with distance:** `validExitsForArmed` returns `[{name, dist}]` sorted ascending by `dist` = the BTV exit distance in metres (raw threshold→exit minus the 400 m touchdown zone == `btvExitDistance`, so the pick-list and the armed-exit readout report the SAME number); the snapshot carries a parallel `btv.exitDists` array, and the form's exit combo items read "Q, 2,289 m / 7,510 ft" (unit per the metric toggle) so the screen reader announces how far down the runway each exit is while arrowing the dropdown. **Rejection feedback:** `EnqueueCommand` captures the arm result and raises `oans_action`; the form announces non-success results (e.g. "Exit K1 not valid for this runway …") so a rejected exit is never silent. ⚠️ NEVER loop `armExit`/`armRunway` over the FULL exit list (~290 calls) — each ACCEPTED arm triggers a BTV canvas redraw and a tight full-sweep exhausts host memory; the validity FILTER above is the read-only way to enumerate valid exits (arming only the 1–2 same-named features, as the fix does, is fine). **Loading an airport: call `oansRef.instance.loadAirportMap(icao)` DIRECTLY, NOT the `oans_display_airport` bus event.** When the OANS is in performance-hide mode (FBW perf mode on + the ND not rendering the OANS — the normal case for us since we never render), the `oans_display_airport` handler (Oanc.tsx) only sets `dataAirportIcao` and SKIPS the load, leaving the ICAO showing with `data===undefined` / 0 labels / `btvReady` false (symptom: "(KLAX)" with no runway list). `agent.displayAirport()` therefore calls `loadAirportMap(icao)` directly (bus pub is the fallback), which loads the AMDB + builds labels regardless of perf-hide (live-verified KLAX → 4789 features / 828 labels, persists; `getAirportData` itself was never the problem). Same gate disables the cockpit auto-load under perf-hide, so the user must press "Display airport" (or Origin/Dest/Alternate) to load — the banner says "maps available — load an airport" until they do.

**Accessible BTV exit arming — native combos + buttons + auto-announce (2026-06, live-verified at VCBI).** On the real A380 you SELECT the BTV landing runway + exit by **clicking the runway-end + exit LABELS on the moving map** (`Oanc.tsx` → `btvUtils.selectRunwayFromOans` / `selectExitFromOans`). Those are SVG on the map canvas with no control-panel widget, so a blind pilot cannot reach them — and selecting RWY/TWY in the control panel only filters the *map*, it does not arm BTV (the user's "if i select rway nothing is shown" / "can't select taxiways after the landing runway is selected" — that dropdown is a map-filter, not an arming control). **`coherent-oans-agent.js` calls the same `btvUtils` methods directly** via the ND instrument instance (`document.querySelector('a380x-nd').fsInstrument.oansRef.instance` → `btvUtils`):
- `oanc()` / `btvReady()`; `btvRunwayList()` / `btvExitList()` (from `labelManager.labels` filtered by `associatedFeature.feattype`: RunwayThreshold=2, runway-centerline=4, RunwayExitLine=19);
- `armRunway(name)` (finds the centerline label + the `data.features` threshold with `idthr===name`, calls `selectRunwayFromOans(icao+name, centerline.associatedFeature, threshold)`); `armExit(name)` (requires a runway armed first; `selectExitFromOans(name, exitLabel.associatedFeature)` — returns "wrong side / too close to threshold" if FBW rejects it); `clearBtv()`;
- `snapshot().btv → `{ready, runways:[], exits:[], runway, exit, lda, exitDist, dry, wet, stop, computing}` — the pick-lists, the armed runway/exit (these are **NOT L:vars**, only OANS observables `btvRunway`/`btvExit`/`btvRunwayLda`/`btvExitDistance`), and the predicted **DRY / WET / live STOP-bar** distances read from the plain-metre L:vars `A32NX_OANS_BTV_{DRY,WET,STOP_BAR}_DISTANCE_ESTIMATED` (0 until the FMS computes the landing on approach; stop-bar only surfaced once dry/wet compute, since it carries a ~40 m ground offset).
`snapshot()` returns the BTV block (data-only — see the zero-render note above). `CoherentNDClient` parses it (`Snapshot.btv`), folds it into the dedup hash, pushes `btv.*` keys, and exposes commands **`oans_arm_runway`** / **`oans_arm_exit`** (value=name) / **`oans_clear`** (plus `oans_display_airport` / `oans_set_manual_stop`). `FBWA380OansForm` builds a **BTV runway combo + "Arm runway"**, an **exit combo + "Arm exit"** (enabled once a runway is armed), and **"Clear BTV"** from the snapshot; the readout shows the armed runway (+ LDA), exit (+ distance), and dry/wet/stop predictions; and it **auto-announces on change** ("BTV armed, runway 22, exit E, 2835 metres" / "BTV cleared"). **Live-verified end-to-end at VCBI** (driving the agent directly): `snapshot().btv returns runways `["04","22"]`, after `armRunway("22")`+`armExit("E")` → `runway:"VCBI22", exit:"E", lda:3351, exitDist:2835`, exits `["A".."E"]`; `clearBtv()` resets to `runway:null`. (Source: `fbw-common/.../OANC/OansBrakeToVacateSelection.ts` + `Oanc.tsx`; selected idents live only on the event bus / observables, so the snapshot reads them from the ND JS instance, not SimVars.)

**OANS refinements (2026-06): units, ROT/turnaround, tab header, tab-aware search.**
- **Distance UNITS follow the metric toggle.** The OANS/BTV distances (LDA, exit distance, dry/wet/live-stop-bar) are stored in METRES but the real ND shows them in **M or FT per the metric setting** (FBW: `CONFIG_USING_METRIC_UNIT` → `UnitType.METER/FOOT`, factor 3.280839895). `snapshot().btv` reads **`A32NX_EFB_USING_METRIC_UNIT`** (the var MSFSBA already follows for weights, which tracks the EFB "US Units" toggle) as `metric`; the form's `Dist()` shows metres or feet accordingly. So OANS distances honour the same unit toggle as the rest of MSFSBA — verified `metric:true` live.
- **BTV ROT + TURNAROUND times (ARINC429-decoded).** `A32NX_BTV_ROT` (runway occupancy time, **seconds**) and `A32NX_BTV_TURNAROUND_{MAX,IDLE}_REVERSE` (brake-cooling time to next departure, **minutes**, idle = hotter brakes = longer) are **ARINC429** words. `coherent-oans-agent.js` `A._arinc(raw)` decodes them (low 32 bits = IEEE-754 float32 via `DataView`, SSM = bits 32-33; valid when SSM 3/2). Surfaced in the readout, **gated on an EXIT being armed** (matching the real OANS, which only shows them then). Live-verified the decode: `rot:30, turnMax:171, turnIdle:197` with an exit armed. Phrasing: "Runway occupancy time 30 seconds" / "Turnaround: 171 minutes max reverse, 197 minutes idle reverse".
- **Current-tab header + tab-aware search.** The readout drops the three tab lines (kept in the controls list to switch) and shows one **"Current tab: MAP DATA"** header derived from the `(active tab)` marker (live-verified clicking a tab moves the active marker). The **search box is tab-dependent** — the "Runway or exit" InputField on MAP DATA is the *airport-code* field on ARPT SEL (with ICAO/IATA/CITY radios) — so the form labels the search box from the live input's own label (`_searchField`, the text before the colon) and disables it when no input is present. `setValue` on the field is live-verified ("ok"); the taxiway search drives the FBW InputField (its accessible payoff is map-centering, so the future airport-diagram reader is the real "tell me about taxiway B" win).

**MFD navigation is via the always-present HEADER, NOT an "INIT hub" — KEEP the page picker (#105).** Investigated the theory that "every MFD page is reachable from INIT" (true-ish on the A320/A330 MCDU via the physical page keys): it is **FALSE for the A380**. Source (`FmsHeader.tsx`) shows every FMS page carries a persistent header of 4 dropdown tabs — **ACTIVE** (F-PLN/PERF/FUEL&LOAD/WIND/INIT), **POSITION** (MONITOR/REPORT/NAVAIDS/IRS/GNSS/TIME), **SEC INDEX** (SEC 1/2/3), **DATA** (STATUS/WAYPOINT/NAVAID/ROUTE/AIRPORT/PRINTER) — plus the top system tabs (FMS/ATCCOM/SURV/FCU BKUP). INIT is just one item under ACTIVE; it has no links to other pages. **Sighted pilots navigate via this header, not through INIT.** Crucially, the MFD scrape does NOT capture the header tabs (verified live: 0 header-tab elements scraped though 40 `_MFD_pageSelector` DOM nodes exist — they're the navigateById click targets, outside the scraped content area), so the **"Go to MFD page" combo is currently the ONLY way a blind pilot can reach most pages** (besides the quick buttons + Ctrl+arrows). It is the accessible equivalent of the always-visible header — strictly better than expanding 4 dropdowns one item at a time. **Do NOT remove the page picker** — it would strand the user. (Confirmed: keep it.)

**ATC COM / CPDLC + D-ATIS — navigate by UIService URI, NOT page-selector id.** The page-selector ids (`{CAPT|FO}_MFD_pageSelector{Tab}_{idx}`) only exist while that *system's* header is mounted, so the FMS-header `navigateById` click can't reach ATCCOM from an FMS page. The reliable cross-system path is the MFD's own **UIService.navigateTo(uri)** — verified live to jump from `fms/data/status` straight to `atccom/msg-record`. The UIService isn't always on `fsInstrument` directly; `A.mfdUiService()` resolves it from `document.querySelector('a380x-mfd').fsInstrument.uiService` OR `…mfdCaptRef/mfdFoRef.instance.uiService` (the mounted side). `A.navigateUri(uri)` (agent) → `navigate_uri` command (`CoherentDebuggerClient`) → `SendNavigateUri` (`FBWA380MCDUForm`). The six ATCCOM pages are in `AllPages` with a non-empty `Uri` field (page-selector dispatch picks `navigate_uri` when `Uri` is set, else `navigate_by_id`); the **ATC COM button** (Alt+A) now fires `SendNavigateUri("atccom/connect")` instead of the old KCCU-only `navigate_by_id("",-1,"ATCCOM")`. URIs (from `AtccomHeader.tsx`): `atccom/connect`, `atccom/request`, `atccom/report-modify/position`, `atccom/msg-record`, `atccom/d-atis/list`, `atccom/emer`. **Live-verified all 6 navigate + scrape as clean DOM text:** CONNECT (NOTIFY TO ATC / ACTIVE+NEXT ATC / DISCONNECT ALL / ADS-C toggles), MSG RECORD (the CPDLC log — ALL MSG / MONITORED MSG, where uplinked clearance text reads when `A32NX_DCDU_ATC_MSG_WAITING` fires), and D-ATIS (DEP/ARR + UPDATE/PRINT) are functional; REQUEST / REPORT & NOTIFY / EMER read "ERROR 404 NOT FOUND" — **WIP in the FBW dev build**, not our bug (will surface once FBW implements them; MSFSBA already navigates to them). The MSG RECORD page being empty with no datalink connection is correct (no CPDLC messages = nothing to read). This is the read path that complements the existing "ATC Message Waiting" auto-announce, which only flagged a message but couldn't read its content.

**Fuel pumps + niche controls + pitch-trim correctness (post-#107 user follow-up).** (1) **Feed-tank FUEL PUMPS** — were entirely missing. Added as Off/On combos in the Fuel panel (`_fuelPumpCircuits` map + a HandleUIVariableSet branch; the calculator-path toggle is live-verified — circuit 2 flips 0↔1). (2) **Pitch-trim correctness** — MSFSBA's "Pitch Trim" read the stock `ELEVATOR TRIM INDICATOR` (normalized −1..1), which is the WRONG representation for the A380. Fixed to read **`ELEVATOR TRIM POSITION` (radians)** and decode in `TryGetDisplayOverride` to "X.X degrees up/down" (positive = UP) — matching the A380 SD `PitchTrim.tsx` (`thsPositionRadians * RAD_TO_DEG`, `Math.sign === 1 ? 'UP' : 'DN'`). The predicted-takeoff-trim readout (`A32NX_TO_PITCH_TRIM`, ARINC429 BNR ±180 = degrees) is correct as-is. The trim WHEEL set isn't a combo (a blind pilot reads the target + actual and adjusts with the global elevator-trim keys/axis — the readouts are the accessibility win).
