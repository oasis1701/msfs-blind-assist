# Variable / Control Troubleshooting Playbook (Universal)

> **Read this when:** a control "doesn't work", reverts, reads a wrong/raw value,
> or seems "computed-output / not modelled" — on ANY aircraft. Before concluding
> a control is broken, follow this method.

## VARIABLE / CONTROL TROUBLESHOOTING PLAYBOOK — UNIVERSAL (read this before saying "X doesn't work")

**This applies to EVERY aircraft in MSFSBA, not just the A380** — the FlyByWire A32NX/A380X, the Fenix A320, the PMDG 737/777, the HorizonSim 787, and any future proprietary add-on. The *mechanisms* differ per aircraft (one uses L:vars, another K-events, another a DOM bridge or a debugger socket), but the **method for finding the mechanism and proving a control works is identical**. The A380 is the worked example below because that's where we learned it the hard way; translate the specifics to whatever aircraft you're on.

**The #1 mistake this project makes repeatedly is declaring a control "broken"/"computed-output"/"reverts"/"not modelled" from a single weak test, then removing or stubbing it — and being WRONG.** It has happened on the chronometer, the Flight Director, the fuel pumps, the ECL window, the wipers, a whole batch of GPWS/recorder switches, and the MFD/KCCU interaction. Every one of those was actually fine; we just tested it wrong or assumed the hard path. The case studies at the end of this section show exactly how each false "it's broken" was overturned, so you can recognise the pattern. So:

**GOLDEN RULES**
1. **Never conclude "doesn't work" from the MCP `set_lvar` tool.** It uses a native data-def write that is UNRELIABLE for FBW L:vars and produces false reverts. ALWAYS test writes via the **calculator/MobiFlight path** — `execute_calculator_code "<val> (>L:<VAR>)"` (MCP) or `simConnect.ExecuteCalculatorCode` (app) or `SimVar.SetSimVarValue('L:VAR','number',v)` in a Coherent view.
2. **A variable can be fully working even with NO audible/visible feedback.** Many switches (GPWS modes, entertainment, NSS, service interphone) toggle and stick but the sim plays no sound and shows nothing a blind pilot notices. "I can't tell if it did anything" ≠ broken. **Confirm by READ-BACK, not by listening.** If `get_lvar` shows the new value held after ~2 s, it works — ship it.
   - **BUT: the inverse trap — a DEAD/OUTPUT-MIRROR L:var passes the read-back yet drives nothing.** Some L:vars are output mirrors the model WRITES (or legacy vars nothing reads). You write `(>L:VAR)`, it holds your value on read-back (rule above seems satisfied!), but no system reacts — because the real control is a *stock simvar* or *input event*, and your L:var is downstream scenery. **So read back the DOWNSTREAM EFFECT, not the L:var you wrote:** the stock simvar it should drive, the bus/light/valve output, the system state. If the L:var holds but the downstream value doesn't move, the L:var is dead — find the real actuator (stock event). Live-burned on this on the A380 ext-lighting (`LIGHTING_STROBE_0`, `A380X_OVHD_EXTLT_NOSE`, `A380X_OVHD_EXTLT_RWY_TURNOFF` all hold writes but the installed Asobo-template lighting ignores them — `LIGHT STROBE/TAXI` never moved; the working path was the stock `STROBES_SET`/`TOGGLE_TAXI_LIGHTS`). The prior session's "verified" nose light was this exact false positive (it read `LIGHT TAXI:2` changing — but that's the runway-turnoff index, coincidence). Conversely, a *mirror* L:var (GEN/APU-GEN `PB_IS_ON`, copied from a stock simvar each frame) makes the stickiness test LIE the other way: writing the value it already holds "sticks" trivially — to test it you must write the OPPOSITE of the current downstream simvar, or just drive the stock event and watch the mirror follow.
3. **Test EVERY state of a multi-position control, not just 0→1.** A 3-position selector (e.g. ANN LT TEST/BRT/DIM, ENGINE MODE CRANK/NORM/IGN, autobrake DISARM/BTV/LO/L2/L3/HI) must be set to each value and read back. Something can stick at 1 but revert at 2, or vice-versa.
4. **Stickiness test:** read original → write target via calculator path → **wait ~1–2 s (several frames)** → read back. Held = settable. Reverted within a frame = genuine computed output. Then RESTORE the original. (An immediate same-tick read is NOT a revert test — computed outputs revert a frame later.)
5. **The reliable existence test is the FBW SOURCE** (locally available — see the user's download path in session context, or clone it): a control var must appear in the cockpit XML/behavior templates AND have a systems-side reader — the write-stick test passes on ANY nonexistent L:var. The 2026-06 dev-source audit found six A32NX controls "verified" that way which were dead (GEN PBs, wipers, dome light, lighting presets, cargo air, ACP volumes).

**WRITE-MECHANISM DECISION TREE** (try in this order; each is live-proven on the A380):
- **Plain FBW L:var** (most controls): `<val> (>L:VAR)`. Works for switches, ND mode/range, EFIS filters, niche toggles. This is the A380 def's catch-all in `HandleUIVariableSet` for `A32NX_`/`A380X_`/`FBW_` keys. (NOT the transponder mode — that is systems-host-managed on the stock `TRANSPONDER STATE:1` and reverts any external write; only squawk code + IDENT are settable. NOT the Flight Director on the A380X — its `A32NX_FCU_EFIS_L/R_FD_ACTIVE` L:var is DEAD there, a write holds but drives nothing; use the stock event below. On the A32NX those same L:vars DO work — this is a per-aircraft divergence, see a32nx.md.)
- **Stock K-event:** `SendEvent("EVENT", param)` or `<param> (>K:EVENT)`. e.g. `XPNDR_IDENT_ON`; the Flight Director is `TOGGLE_FLIGHT_DIRECTOR` with the SIDE as the param (1 = Capt, 2 = F/O — live-verified per-side; param 0 only ever turns it off), state read from the stock `AUTOPILOT FLIGHT DIRECTOR ACTIVE:n`.
- **Indexed K-event:** `SendEvent("EVENT:index", val)` or multi-param `<a> <b> (>K:N:EVENT)` where N = arg count. e.g. engine anti-ice `ANTI_ICE_SET_ENGn`, ignition `TURBINE_IGNITION_SWITCH_SETn`.
- **H-event:** `(>H:EVENT)` — for things wired only to HTML events. e.g. the clock CHR: `(>H:A32NX_CHRONO_TOGGLE)` / `(>H:A32NX_CHRONO_RST)` (NOT the EFIS chrono K-event — wrong button). Verified the elapsed-time L:var then advances.
- **Electrical CIRCUIT controls** (wipers, feed-tank fuel pumps, emergency exit lights, some lights): the FBW behaviour XML drives a circuit, not a plain L:var. On/off = `<circuitId> (>K:ELECTRICAL_CIRCUIT_TOGGLE)` (state read from `A:CIRCUIT SWITCH ON:<id>`); power/speed = `<percent> <circuitId> (>K:2:ELECTRICAL_CIRCUIT_POWER_SETTING_SET)` (read `A:CIRCUIT POWER SETTING:<id>`). Fuel pumps use `<id> 1 (>K:2:ELECTRICAL_BUS_TO_CIRCUIT_CONNECTION_TOGGLE)` with `A:CIRCUIT CONNECTION ON:<id>`. **Find the circuit id in the behaviour XML** (`#CIRCUIT_ID_…#` in `model/behaviour/legacy/generated/A32NX_Interior_*.xml`), don't guess.
- **FCU values:** the A380 has NO `A32NX_FCU_AFS_DISPLAY_*` vars; use the `A32NX.FCU_*_SET` events (heading/speed/alt) and read the raw `A32NX_AUTOPILOT_*_SELECTED` SI-unit vars (convert rad→deg, m/s→kt/fpm). Signed values (V/S, FPA) must go via calculator code, not `SendEvent` (uint overflow).
- **Genuine computed outputs** (rare): a handful of overhead PBs are recomputed every frame by the FBW systems (some bus-ties). Only after the stickiness test ABOVE reverts on EVERY path do you treat it read-only — keep the read+announce, drop the set.

**WHERE TO DIG when a mechanism is unknown** (do ALL of these before giving up):
1. The live L:var catalog: `tools/_probe/fbw_lvars_clean.txt` (every L:var in the running aircraft) — grep the system name and its synonyms (e.g. wiper → also RAIN, REPELLENT).
2. The FBW **behaviour XML**: `fbw-a380x/.../model/behaviour/` — the cockpit node templates show the EXACT K/H/L mechanism and circuit ids (`awk '/Template Name="…"/,/<\/Template>/'`).
3. The **Rust systems** crate `fbw-common/src/wasm/systems/` and the **instruments** `fbw-a380x/src/systems/instruments/` (TSX) — for what publishes/consumes a var and its units (e.g. reservoir level = gallons, oil = quarts).
4. Live probe with `tools/coherent-eval.ps1 -Title A380X_SYSTEMSHOST -ExprFile <probe.js>` (reads/sets L:vars + stock simvars in-context) and the **simconnect MCP** (`get_lvar`/`get_simvar`/`execute_calculator_code`/`trigger_event`/`search_lvars`).
5. The FBW docs (docs.flybywiresim.com) for intended behaviour.

**TOOLS that exist in this repo:** `tools/coherent-eval.ps1` (Coherent eval), `tools/_probe/*.js` (probes + README), `tools/fbw_lvars_clean.txt` (catalog), the `fbw-aircraft` sparse clone at `C:/Users/franc/Documents/development/fbw-aircraft`. The simconnect MCP is the fastest write-stick tester.

**Captain/First-Officer rule:** where a control GENUINELY exists per side AND both sides work, expose BOTH as a split `… Captain` / `… First Officer` panel — never one shared control, and never invent a First-Officer var that doesn't exist. Verify each side's var is real (reads a value, not ERR) and settable before adding it. (A380 examples: EFIS, baro, FD, ND mode/range, navaids, QNH, and the RMP-1/RMP-2 audio panels.)

**GENERALISING TO OTHER AIRCRAFT.** The decision tree and "where to dig" above are A380/FBW-flavoured, but the *shape* is universal — for any aircraft, the job is (1) find the variable/event the cockpit actually uses, (2) find the write mechanism that sticks, (3) prove it by read-back. Where to look depends on the add-on's architecture:
- **FBW (A32NX/A380X):** open-source — L:var catalog, the behaviour XML, the Rust `systems` crate, the instruments TSX (everything in the `fbw-aircraft` clone). Writes via the MobiFlight calculator path or K/H events.
- **Proprietary with a SimVar/L:var surface (Fenix, PMDG):** no source — discover vars empirically. PMDG exposes a documented data struct + custom events (`send_pmdg_event`, `get_pmdg_var`); Fenix uses L:vars settable via the native data-def path (the one case where data-def IS the right write). Use the MCP `search_lvars`/`list_lvar_*`/`get_pmdg_*` to enumerate, then the stickiness test.
- **DOM/web add-ons (the A380 MFD/flyPad/ECL, any WebView/Coherent UI):** there's no SimVar — the control is in a rendered document. Read/drive it through the debugger socket (`coherent-eval.ps1`, `CoherentEWDClient`) or the WebView. Two hard-won rules: **Coherent GT (Chromium 49) allows only ONE inspector socket per page** (share the connection — that's why the ECL "showed nothing": a second socket was rejected), and a control may have **no DOM click handler** (the ECL is driven by the ECP push-button L:vars, not by clicking the lines; the MFD by the FBW `InputField` keypress path, not the KCCU cursor H-events). When a UI "can't be clicked", find what the real cockpit input actually drives underneath — don't assume the obvious element is the lever.

**CASE STUDIES — false "it's broken", and the reasoning that overturned each** (recognise these patterns):
- **Chronometer ("not modelled"):** first test pulsed the wrong control (the EFIS/ND chrono K-event) and the elapsed-time L:var didn't move → wrongly "not modelled". Reality: the clock CHR runs off the **H-event** `A32NX_CHRONO_TOGGLE` (gated on DC ESS power); fired that, the L:var advanced (10.8 s after toggle). LESSON: a var not changing means *your trigger* might be wrong, not that the var is dead — find the actual input in the source.
- **Flight Director (the stickiness trap, twice):** an earlier pass wrote the engage L:var `A32NX_FCU_EFIS_L/R_FD_ACTIVE` via the calculator path, saw it HOLD (set 1 → still 1 after 2.5 s), and concluded the FD was controllable. Reality (2026-07, live-verified in flight): that L:var is **DEAD** — the FD stayed ON with the L:var reading 0 (fully decoupled), and writing it actuated nothing. The write held but drove nothing — the deepest form of the stickiness trap: a hold is necessary but NOT sufficient. The real control is the cockpit FD button's `K:TOGGLE_FLIGHT_DIRECTOR` with the SIDE as the param (1 = Capt, 2 = F/O), state on the stock `AUTOPILOT FLIGHT DIRECTOR ACTIVE:n`. LESSON: a write that sticks is only step 1 — you must also confirm a DOWNSTREAM effect (the FD indicator/mode actually changed), or a dead L:var will pass the stickiness test forever.
- **Fuel pumps ("reverts/uncontrollable"):** the MCP `set_lvar` (data-def) appeared to revert → wrongly "computed output". Reality: feed-tank pumps are electrical **circuit connections** toggled with `(>K:2:ELECTRICAL_BUS_TO_CIRCUIT_CONNECTION_TOGGLE)`; via the calculator path they stick. LESSON: never conclude from `set_lvar`; and some "switches" are circuits, not plain L:vars.
- **Wipers ("completely inop"):** the code only set circuit *power* and never *toggled the circuit on*, so they never started. Reality (from the behaviour XML): on/off is `<id> (>K:ELECTRICAL_CIRCUIT_TOGGLE)`, speed is `<pct> <id> (>K:2:ELECTRICAL_CIRCUIT_POWER_SETTING_SET)`; both verified live. LESSON: read the cockpit node template for the EXACT sequence — a partial write looks like a dead control.
- **ECL window ("shows nothing"):** the failure monitor already held the one allowed Coherent inspector socket, so the checklist's second connection was silently rejected. Reality: share the single socket. LESSON: for Coherent UIs, suspect the one-socket-per-page limit first.
- **GPWS / recorder / NSS / entertainment ("unreliable / can't tell if it works"):** write-stick tested all of them — every one set and held after 2 s. They were never broken; the sim just plays no sound and shows nothing a blind pilot notices. LESSON: "I can't tell" is not data — read back the value.
- **MFD / KCCU interaction ("needs the cursor / can't be driven"):** assumed the MFD needed the KCCU trackball/cursor and the ECL needed DOM clicks. Reality (from the instrument source): the MFD takes text via the FBW `InputField` (focus-click + `keypress` with keyCode, Enter=13) and the ECL is driven by the ECP button L:vars — neither needs the KCCU cursor. LESSON: trace what the real hardware input writes; the visible widget is often not the thing you poke.
- **Ext lighting Strobe/Nose ("verified working" — but DEAD):** the def drove `LIGHTING_STROBE_0` and `A380X_OVHD_EXTLT_NOSE` as L:var combos; both *hold* writes, so a naive L:var read-back "passed". Reality: the installed A380X uses Asobo stock-simvar lighting templates — those L:vars are output-only/legacy and appear nowhere in the FBW source; writing 0/1/2 left `LIGHT STROBE:1`/`LIGHT TAXI:*` unchanged. The working actuators are the stock `STROBES_SET` and `TOGGLE_TAXI_LIGHTS` events (live-verified to move the simvars). The prior "verified" nose-light note read `LIGHT TAXI:2` (the runway-turnoff index, not nose) changing — a coincidence. LESSON: read back the DOWNSTREAM simvar, not the L:var you wrote; an L:var that holds but drives nothing is dead. (`A380X_OVHD_EXTLT_RWY_TURNOFF` is dead the same way — left UNexposed.)
- **GEN / APU-GEN pushbuttons ("stickiness says settable"):** `A32NX_OVHD_ELEC_ENG_GEN_n_PB_IS_ON` / `_APU_GEN_i_PB_IS_ON` are COMPUTED MIRRORS the FBW WASM copies from stock `GENERAL ENG MASTER ALTERNATOR:n` / `APU GENERATOR SWITCH:i` each frame. Writing the value they already hold "sticks" → the stickiness test falsely says settable, but a real off-command is overwritten next frame. The working control is the stock event (`TOGGLE_MASTER_ALTERNATOR` engine-index for ENG GEN; indexed `APU_GENERATOR_SWITCH_SET` for APU GEN) — verified by toggling and watching BOTH the stock simvar AND the L:var mirror follow. LESSON: when an L:var is a mirror of a stock simvar (find the `.copy(aircraft(...), aspect(L:var))` in the Rust `*_systems_wasm/src/lib.rs`), drive the stock event; keep the mirror for read/announce.
- **HYD electric-pump A/B ON/OFF ("reverts to AUTO"):** the green/yellow EMP A/B `_ON_PB_IS_AUTO` / `_OFF_PB_IS_AUTO` are toggled by the cockpit (`TOGGLE_SIMVAR`) but the FBW `A380ElectricPumpController` returns them to AUTO when the backup pump isn't needed (on the ground, EDPs pressurising). So a calc write reverts — but that is CORRECT conditional behaviour, not a dead var (the ENG-driven pumps and the two MAIN electric pumps `EPUMPB`/`EPUMPY` hold a manual command and are fully settable — the difference is the per-pump A/B controller logic). Keep them settable (probe-heat precedent); the auto-announce re-reads true state. Also: the ON button is `On/Auto` (0 = forced ON), the OFF button is `Off/Auto` — don't label the ON button OffAuto (selecting "Off" would force it ON). Each EMP genuinely has TWO physical pushbuttons, so two combos is faithful — not a unify candidate.
- **Annunciator / integral LT TEST is render-only — do NOT synthesise:** setting the ANN/INT LT knob to TEST illuminates every overhead light in the SVG but changes NO L:var (live-verified: no `_PB_HAS_FAULT`/annunciator var moves). There is nothing real to announce, so the knob just speaks its position (Test/Bright/Dim) and the real per-system fault lights announce when a genuine fault appears. LESSON: MSFSBA announces real state; if a "test"/action has no readable consequence, announce the control's own position, never a hand-written narration of what the bulbs would show.
- **Every panel control is a COMBO, not a hardware button** (screen-reader principle, user-mandated): a momentary action (TEST / ACK / RESET / chrono / rain-repellent) renders as an `Off`/`Activate` (or `Idle`/`Activate`) combo whose "Activate" option fires the pulse/H-event in `HandleUIVariableSet` (guarded on `value > 0.5`); not auto-announced (the handler speaks "<name> pressed"). The only remaining true buttons are the FCU momentary knob push/pulls (genuine knob actions with post-press value read-outs via `OnPanelButtonFired`) and value-entry fields (squawk code) — those aren't on/off switches.

**Pitch trim (`ELEVATOR_TRIM` panel display) reads `ELEVATOR TRIM POSITION` in DEGREES**, the same unit as the shared base `MON_ElevatorTrim` (Shift+T continuous trim-announce) — so no same-simvar unit mismatch. Decoded to "X.X degrees up/down" (positive = UP, matches the A380 SD PITCH TRIM block). The Shift+T trim-announce runs entirely off the base var and is unchanged on the A380 and every other aircraft (the A380 panel display is OnRequest-only and does not touch it).

**Variable provenance.** The A380X reuses the `A32NX_` prefix for almost all systems (index ranges extended: 4 engines/gens/IDGs/bleeds/FADECs, 11 fuel tanks, 16 brakes, 4 outflow valves). `A380X_` is reserved for genuinely new hardware (RMP suite, EFIS-CP/ND, fuel transfer/jettison, battery display knob, ANN/INT-LT, OIT, storm light, MFD page URIs). Source completeness from the FBW **A32NX docs** (docs.flybywiresim.com) too — MSFSBA's own A320 definition is incomplete, so don't treat it as the ceiling. Reference catalogs in `tools/`: `a380-simvars-catalog.md`, `a380-fcu-vars.md`, `a380-engine-start.md`, `a380-fault-status-vars.md`, `a380-sd-pages.md`, `fbw-a380-mfd-source-findings.md`, and the live L:var dump `_probe/fbw_lvars_clean.txt`. FCU mapping (the A380 has NO `A32NX_FCU_AFS_DISPLAY_*`/`*_LIGHT_ON` vars) is in `a380-fcu-vars.md`.

**FCU is ported from the A320** (set dialogs + paired readouts via `HandleHotkeyAction`; A380 autoflight vars). Not out of scope — it reuses the A320 `A32NX.FCU_*_SET` events with A380 value/managed/mode vars.

**A380 FCU windows (Fenix-style, input mode).** Ctrl+S/H/A/V/P/B open dedicated `Forms/FBWA380/FBWA380{Speed,Heading,Altitude,VS,Autopilot,Baro}Window` forms (shared base `FBWA380FCUWindowBase` — non-modal `ShowForm()`, focus capture/restore, Escape-to-close), replacing the old bare `ValueInputForm` dialogs (`ShowFCU*Dialog`/`ShowA380BaroSetDialog` deleted). Each form is thin UI delegating to public `FlyByWireA380Definition` methods — `SetFCU{Heading,Speed,Altitude,VS}Value`, `FireFCUButton(evt)` (Push/Pull/SPD-MACH/TRK-FPA, readback via `OnPanelButtonFired`), `RequestFCU*WithStatus` (spoken live readout on open + after each action), `RequestAutopilotStates`, `ToggleMetricAltitude`, `SetAltIncrement`, and `ApplyUIVariable` (a `HandleUIVariableSet` passthrough that looks up the `_varCache` def) — no FCU logic in the forms. The **Baro** window enters QNH ONCE and applies to both altimeters (`CAPT_QNH_SET` → `KOHLSMAN_SET` moves both); the STD/QNH toggle fires `H:A380X_EFIS_CP_BARO_PULL/PUSH_{1,2}` with state from the stock `KOHLSMAN SETTING STD:n` (the `*_EIS_BARO_IS_STD` keys remain as cache keys only — dev FBW removed those L:vars); the hPa/inHg toggle syncs both sides via `XMLVAR_Baro_Selector_HPA_{1,2}` (dev FBW honors `_1` only). The window is Fenix-style (Mode/Unit combos + value box, 2026-06). The **Autopilot** panel (Ctrl+P, previously unhandled on the A380) covers AP1/AP2, A/THR engage (`AUTO_THROTTLE_ARM`) + A/THR + AP disconnect, APPR/LOC/EXPED, with button-state labels refreshed from the cache and **FD read-only** (uncontrollable on this build). Per the screen-reader rule the windows never re-announce button presses — only value-set confirmations + readouts speak. Verify live with `tools/fcu/fcu-read.ps1` (full FCU state) / `fcu-roundtrip.ps1`; note probe-side `A32NX.FCU_*` sets are no-ops (set via the app, read back via the harness).

**FCU readouts need SI→display conversion (the A380 has NO `A32NX_FCU_AFS_DISPLAY_*` vars).** The A320 reads those pre-formatted display vars; the A380 lacks them, so MSFSBA reads the raw `A32NX_AUTOPILOT_*_SELECTED` targets, which the SimConnect data-def read returns in **SI base units**. `ProcessSimVarUpdate` converts: heading/track radians→deg (`Math.Abs(v)<=2π ? v*180/π`), FPA radians→deg (`<=0.2 ? *180/π`), V/S m/s→fpm (`*196.8503937`, round to 100), speed m/s→kt (`*1.943844`). Altitude is a SimVar read in feet already. Symptom if a conversion is missing: V/S reads "15" instead of 2000, heading "004" instead of a real bearing. FCU set/readback (`Shift+A/H/S/V`), V/S sign, and the managed-mode dot are all live-verified.

**Flight director IS controllable — settable per-side combos (RE-CORRECTED 2026-07, live-verified).** The A380 FD is exposed as "Flight Director 1/2" combos. **The actuator is the stock `K:TOGGLE_FLIGHT_DIRECTOR` with the SIDE as the PARAMETER (1 = Capt, 2 = F/O), and the state is the stock `AUTOPILOT FLIGHT DIRECTOR ACTIVE:n`** — this is what the def uses and it is live-verified (2026-07: FD1 was ON / stock `:1`=1, fired `1 (>K:TOGGLE_FLIGHT_DIRECTOR)` → `:1` flipped to 0 while `:2` stayed 1; per-side confirmed). ⚠️ **`A32NX_FCU_EFIS_L/R_FD_ACTIVE` is DEAD on the A380X** — it read **0 while the Capt FD was actually ON** (fully decoupled from the real state) and writing it drives nothing (the "holds via the calc path" note was the write-stick trap — it holds but actuates nothing). An earlier note here claimed `_FD_ACTIVE` was the correct var — that was WRONG; do NOT switch the FD combos back to it. (The earlier `TOGGLE_FLIGHT_DIRECTOR` "failure" used param 0, which only ever turned the FD off; the SIDE param is the fix.) `AUTOPILOT FLIGHT DIRECTOR ACTIVE` (combined, no index) is kept as an extra read-only status. FD bars themselves are visual-only, but the engage toggle is real — don't rip out the combos.

**EFIS baro is decoded for both speech AND the panel display.** `A32NX_FCU_{LEFT,RIGHT}_EIS_BARO_HPA` is an ARINC429 word (raw ~14 billion — the "altimeter 14 million" bug). The decoded value auto-announces in `ProcessSimVarUpdate` on knob turn and on STD↔QNH and hPa↔inHg unit changes, is queryable via the altimeter hotkey, AND now reads cleanly in the EFIS Captain/First Officer panels via **`TryGetDisplayOverride`** (see below) — "1013 hPa" / "29.92 inHg" / "Standard", never the raw word. Set-QNH routes through `KOHLSMAN_SET` (param = millibars×16), unit-aware. Minimums (`FM1_MINIMUM_DESCENT_ALTITUDE` / `_DECISION_HEIGHT`) likewise ARINC-decoded for both speech and the **Minimums** panel ("200 feet" / "Not set").

**EFIS baro UNIT (hPa/inHg) comes from `XMLVAR_Baro_Selector_HPA_{1,2}` (1=hPa, 0=inHg), NOT `A32NX_FCU_EFIS_*_BARO_IS_INHG`.** The IS_INHG var is stuck at 0 on the A380X (verified live via the Coherent debugger: the F/O reads `XMLVAR_Baro_Selector_HPA_2 = 0`/inHg while `A32NX_FCU_EFIS_R_BARO_IS_INHG` stays 0/hPa). The combo always wrote the XMLVAR, but the announce/display read the stuck IS_INHG — so a pilot who selected inHg kept hearing hPa. `_baroInHgL/R` is now tracked off the XMLVAR in `ProcessSimVarUpdate` (inHg when `value < 0.5`); IS_INHG is no longer registered. This feeds `BaroPhrase` and `TryGetDisplayOverride`.

**Announcement-quality patterns (2026-05 quality pass).**
- **ARINC429 enum guard.** Some FBW discretes are written `write_arinc429`, so an announced enum var arrives as a huge SSM-encoded word (e.g. `A32NX_APU_LOW_FUEL_PRESSURE_FAULT` = 12884901888 = 0x3_00000000 = SSM NormalOp, payload 0) that matches no 0/1 entry → the generic announce printed "<label>: 12884901888". `ProcessSimVarUpdate` opens with a generic guard: any ANNOUNCED enum var (`_varCache` lookup, `IsAnnounced`, `ValueDescriptions.Count>0`) whose raw value is ARINC-large (`>= 2^32`) is decoded via `Arinc429Word.ValueOr(0)` to its 0/1 payload, announced via its `ValueDescriptions` ON CHANGE only (default prev=0 so the initial no-fault is silent), honouring the Ctrl+M mute, and returns true. Future-proof for every such discrete. (Announced ARINC NUMERICS — EIS baro, minimums — are handled by their own `MonNum` + custom decode, so they have no ValueDescriptions and skip this guard.)
- **Annunciators are NOT panel read-out fields.** `GetPanelDisplayVariables` ends with a filter that strips every ANNOUNCED 2-state enum var (faults + non-fault binary lights) from the display sets — a pilot navigates a panel to operate CONTROLS, not to scan "X Fault: Normal" rows. **Crew SEATS are 8 toggle BUTTONS (press to start, press again to stop + speak position) — 2026-06, user request, supersedes BOTH the b065970 "ramp the slider" AND the intermediate 3-state-combo motor design.** Two earlier bugs drove this: (1) the "just a tick, no sustained motor sound" — the seat motor audio (`sound.xml` `seatmotor{fwd,back,up,down}`) is a Wwise **`Continuous` loop gated by a `Derived="true"` RTPC = the per-frame DERIVATIVE of the position L:var** (`SEAT_{CPT,FO}_MOVE_{UP_DOWN,FWD_AFT}`, 0–100); a write must change the var EVERY frame or the loop only plays its onset then fades (`FadeOutTime=0.1`). (2) the combo "start/stop/start + spurious down/up 5%" — MainForm periodically re-reads a combo's backing var; the synthetic motor var reads 0 → snaps the combo to "Stopped" → re-fires `HandleUIVariableSet` → restart + bogus announce. **Fix = `RenderAsButton` toggle buttons** (MainForm fires `HandleUIVariableSet(varKey, 1, …)` on the click edge only, no periodic re-read, so the snap-back can't happen). Eight buttons `SEATBTN_{CPT,FO}_{UP,DOWN,FWD,AFT}` (`SeatBtn` helper) + four read-only position vars `SEAT_{CPT,FO}_MOVE_{UP_DOWN,FWD_AFT}` (`SeatPos`). `_seatButtonMap` maps each button → `(PosVar, Dir ±1)`; the HandleUIVariableSet branch calls `ToggleSeatMotor`: if that axis is already moving this dir → stop + `AnnounceSeatPosition` (+ `RequestVariable` forceUpdate); else seed `_seatMotorPos` from the cache and start a **20 ms** UI-thread `SeatMotorTick` firing a UNIQUE clamped read-modify-write via the calc path `<seq> 0 * (L:VAR) 0.4 + 100 min (>L:VAR)` (up) / `0.4 - 0 max` (down) — the `<seq> 0 *` prefix makes each calc string unique so MobiFlight doesn't DEDUP it (identical strings are registered+fired once; that dedup was the *real* "tick once" cause), and the per-frame change keeps the derivative non-zero → sustained motor noise. **Position read-out on stop** speaks a BAND not a percentage: `SeatBand` → fully up/up/slightly up/mid travel/slightly down/down/fully down (+ "{pct} percent"); `TryGetDisplayOverride` renders the 4 position vars the same way. 8 s safety cap stops a forgotten-moving axis; at the 0/100 clamp the writes no-op and the loop fades on its own. **Direction labels (Up=+var / Forward=+var) NOT visually confirmed** — if a seat moves the wrong way, swap the Dir sign in `_seatButtonMap` (trivial). **Armrests / sunshades / visors KEEP the `RampSliderTo` slider** — no motor sound in `sound.xml`, so a position target is the right model there.
