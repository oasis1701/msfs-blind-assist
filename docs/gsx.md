# Access GSX

Access GSX is the screen-reader-friendly interface for GSX Ground Services Pro. It mirrors the GSX in-sim menu and tooltip stream in standard Windows controls, so NVDA, JAWS, and other screen readers can read GSX status without relying on the simulator overlay.

## Opening And Reading GSX

- Open the Access GSX window with Input mode > `Alt+G`.
- Press `F5` inside the Access GSX window to open or reopen the GSX menu.
- Choose GSX menu options with `1` through `9`, `0`, then `A` through `E` for options above ten.
- Press `Esc` to hide the Access GSX window. The GSX service keeps running in the background.
- Press Output mode > `Ctrl+G` to read the latest cached GSX tooltip without opening the Access GSX window.

The Access GSX window has separate read-only fields for connection status, menu options, and the latest tooltip. When the GSX menu is hidden or times out, the menu field shows a prompt telling the user to press `F5` again.

## Settings

Press `C` inside the Access GSX window while a GSX menu is available to open the accessible GSX Settings window. The settings window reads GSX's published settings page, groups controls by GSX section, and exposes toggles, choices, ranges, text fields, action buttons, and read-only information with standard Windows controls.

Setting changes are sent to GSX and persisted to the GSX configuration file. Numeric and text settings are committed as they change and again when the settings window closes, with duplicate writes suppressed.

If GSX has not published its settings page yet, open the GSX menu with `F5`, then press `C` again.

## Active Services

When GSX has two or more services running at the same time, the Access GSX window shows an Active services combo box above the tooltip. Selecting a service chooses which active row drives the tooltip and automatic announcement stream. The selector is hidden when zero or one service is active so it does not clutter the tab order during normal use.

## Automatic Announcements

Access GSX speaks meaningful changes from the tooltip/status feed instead of every small text refresh. It handles:

- Boarding and deboarding progress, with repeated progress suppressed.
- Refuel, baggage, catering, and pushback status.
- Persistent ground connections such as GPU, PCA, and chocks, including periodic timer-only status announcements.
- Completed-service totals.
- Invoices and detailed receipts when GSX exposes receipt data.

The Access GSX window speaks announcements while it is visible. When the window is hidden, background tooltip announcements follow the user's Announcement Settings option for GSX background monitoring.

## Implementation Notes

- `Forms/AccessGSXForm.cs` owns the accessible window, menu shortcuts, active-services selector, and settings-window launch.
- `Forms/GsxSettingsForm.cs` renders editable GSX settings from parsed GSX HTML metadata.
- `Services/GsxService.cs` owns GSX menu/tooltip communication, active service selection, announcement throttling, invoice/receipt parsing, and settings persistence.
- `Settings/UserSettings.cs` stores the app-level `GsxBackgroundMonitoring` option; GSX's own settings are persisted to GSX configuration.

## Developer internals

The sections above document **Access GSX** — the accessible GSX menu/tooltip/settings window (PR #84's territory: live SERVICE STATE — boarding/refuel/catering/deice/pushback status, timers). The rest of this document covers a **separate, coupled feature set** layered on taxi guidance (branch `feat/gsx-taxi-gate-integration`): GSX gate/stand **positioning** (selecting a GSX parking position by driving its in-sim menu), audio **docking guidance** to the stop position (VDGS/marshalling analog), and an app-wide metres/feet **distance unit** toggle. These notes are developer-facing implementation detail, folded in verbatim from CLAUDE.md (Task 1.3, 2026-07).

**Domain boundary with PR #84 ("Fix GSX active service") — do not cross.** The docking/positioning code owns POSITIONING only: gate/stand selection, docking geometry, deice-pad positioning. PR #84 owns live SERVICE STATE (boarding/refuel/catering/deice/pushback status, timers, `GsxSettingsForm`). Don't add service-state logic to the docking code, and don't let docking read service vars.

**Feature:** Three coupled subsystems layered on taxi guidance — (1) selecting a GSX parking position by driving its in-sim menu, (2) audio docking guidance to the stop position (VDGS/marshalling analog), and (3) an app-wide metres/feet distance toggle. Verified with console probes under `tools/` plus xUnit characterization tests (`DockingGeometryTests`, `GsxOffsetTests`) run by CI (run `dotnet run --project tools/<Name> -p:Platform=x64`): `DockingProbe`, `DistanceUnitsProbe`, `GsxProfileProbe` (`--all` sweep / `--audit` robustness), `GsxAirplaneProbe`.

**Key rules when touching this code:**

- **Gate-list source selection lives in `Services/GateDataSource.cs`.** GSX is authoritative when GSX is available (`CouatlStarted`) AND `GsxProfileLocator` finds a matching profile; otherwise navdata. `GsxNavdataMerger` overlays GSX metadata onto the navdata skeleton (navdata supplies positions GSX omits, matched by concourse+number). **Merger safety: no cross-concourse coordinate borrowing.** When a GSX gate HAS a concourse, a navdata candidate may only donate coordinates if its normalized concourse matches — otherwise the spot is DROPPED (routing a blind pilot to "A12" at B12's coordinates on another pier is worse than omission). **Spot position priority is `this_parking_pos` → navdata → stop position (LAST resort).** The GSX stop position is a VDGS NOSE-STOP reference, not an aircraft-datum location — using it as the spot position teleports the datum metres deep into the stand (sometimes at heading 0); docking is unaffected because it reads the preserved `StopLatitude/StopLongitude/StopHeading` fields directly. **GSX Python (`.py`) profiles are NOT parsed for the gate LIST** — only `.ini`. So large airports with a `.py` profile (e.g. EDDF `eddf-Aerosoft.py`) fall back to navdata for the LIST, even though the live GSX *menu* (used by the selector below) still works. Deice areas are GSX-only and excluded from the normal gate list (`GetDeiceAreas` is separate). **The DOCKING STOP, however, DOES consume `.py`** — see the per-aircraft stop-offset bullet below.
- **GSX `.py` per-aircraft stop offset applies to ALL non-deice gates (incl. `.ini`).** `TaxiAssistForm.ApplyGsxStopOffset` resolves GSX's per-aircraft longitudinal/lateral stop offset (metres) and feeds it to `DockingGuidanceManager.SetStopOffset` for every non-deice gate — `.ini` gates included. It previously SKIPPED the `.py` `customOffset` whenever the gate had a `StopLatitude` (i.e. `.ini` gates), wrongly assuming the `.ini` stop was already aircraft-exact. It isn't: the `.py` `customOffset` is GSX's PER-AIRCRAFT adjustment layered ON TOP of the static `.ini`/navdata base (same gate → different offset per airframe — EDDF A66: 777 = 5.3 m, A380 = 6.3 m, base = 1.65 m). Without this the 777 parked ~5.3 m short at every `.ini` airport (EDDF etc.). **Suffix changes the resolved function:** EDDF **A66** (no suffix) → 777 = **+5.3 m**; EDDF **A66A** (suffix "A") → 777 = **0 m** (that stand's function has no 777 table entry, so it correctly falls to base 0; an A320 there gets −2.5 m — proves it evaluates rather than parse-missing). Locked by `tools/GsxOffsetProbe` resolver-level asserts. A `STOPOFFSET` diagnostic line (icao / gate# / suffix / stopLatSet / ac / acId / resolved offset) is appended to `%APPDATA%\MSFSBlindAssist\logs\docking-aircraft.log` on each route-calculate for one-glance debugging of "offset is 0" reports. `GsxStopOffsetResolver` locates the airport `.py` (`GsxProfileLocator.TryFindPyProfile`, excludes `_handler.py`), parses it (`GsxPyProfileReader`, cached by path+last-write), and evaluates the gate's `customOffset` function (`GsxPyOffsetEvaluator`) for the resolved `GsxAircraftId`. **Aircraft resolution is UNIVERSAL** (`GsxAircraftIdMap.TryResolve(icao, wingspanMetres)`): idMajor/idMinor are DERIVED from the ICAO designator pattern (Boeing `B7Xd`→707+X·10, Airbus `A3YZ` narrowbody-literal/widebody-300+Y·10, Embraer literal) and ARC from wingspan (Annex-14 A&lt;15…F≥65 m); a thin exception table holds only irregulars (B787 bare-minor, A350 idMinor 1000, neo idMinor 1, A220, **and the 737 MAX family — `B37M`/`B38M`/`B39M`/`B3XM` → 737 family with closest-gauge minor**: the `B3xM` designators break the `B7Xd` pattern and used to resolve idMajor 0, silently losing every idMajor-keyed `.py` stop offset for one of the most common airliner families; locked by probe asserts). The raw ICAO is always kept so ICAO-keyed tables hit; idMajor 0 forces the base offset. Evaluator group fallback tries `"ARC-E"`, bare `"E"`, and `"Heavy"` (authors key differently; `"ARC-X"` dominates installed profiles). In `DockingGuidanceManager.UpdatePosition` the stop point is shifted (`ShiftStop`, equirectangular) BEFORE any distance is computed — longitudinal along `StopHeading`, lateral perpendicular (right=+). **`GsxOffset.Zero` is a strict no-op** (shift skipped) so behaviour is byte-identical to today; any miss at any layer (no profile/unknown aircraft/parse error) degrades to `Zero`. Logged as `stopOffL`/`stopOffLat` in `docking.log`. Measured golden: EDDF A66 / 777-300 = +5.3 m. Verified by `tools/GsxOffsetProbe` (golden + `--sweep` over all 72 profiles + a **universal installed-aircraft scan**: `AircraftCfgCatalog.EnumerateInstalled()` enumerates THIS machine's real fleet and asserts every distinct ICAO resolves without throwing — zero hardcoded expected values, so the probe works on any user's machine).
- **TITLE→ICAO catalog fallback (`Services/AircraftCfgCatalog.cs`).** The whole GSX/docking aircraft-resolution chain keys on the ICAO from SimConnect `ATC MODEL` (`SimConnectManager.ExtractIcaoFromAtcModel`). For the rare add-on whose ATC MODEL doesn't parse to a clean ICAO, `SimConnectManager` falls back to `AircraftCfgCatalog.TryGetIcaoByTitle(currentAircraftTitle)`. The catalog is a PURE, dependency-light service (file IO only — no WinForms/SimConnect/EFBModPackageManager deps, so the probe links it) that scans installed `aircraft.cfg` files under `<InstalledPackagesPath>\{Community,Official\OneStore,Official\Steam}\…\SimObjects\Airplanes` (depth-bounded so it never crawls texture trees), parsing each cfg's package-level `icao_type_designator` and every `[FLTSIM.N] title` into a `titleLower→icaoUpper` map. It resolves the packages root itself by reading `InstalledPackagesPath` from the FS2024/FS2020 `UserCfg.opt` locations. **Background + lazy + cached + thread-safe; NEVER throws** (any missing dir / locked file / malformed cfg degrades to empty/no-hit). The fallback runs ONLY when `ExtractIcaoFromAtcModel` returns empty, NEVER blocks the SimConnect callback (the scan/lookup is on a `Task.Run`), validates the result looks like an ICAO (2–4 alphanumerics), and — mirroring the door-offset background re-fire — sets `CurrentAircraftIcaoType` + re-fires `AircraftIcaoTypeDetected` when it later resolves. The common case (ATC model present) is byte-for-byte unchanged. The catalog is warmed via `BeginBuild()` at `Connect()` time so the fallback has near-zero latency.
- **`Services/Gsx/GsxGateSelector.cs` drives the live GSX menu via L-vars** (NOT the parsed profile) to select a stand: a bounded, backtracking, **page-aware DFS** (`TraverseAsync`). `ChooseAsync` is only ever called while the live menu is displaying the page containing the entry (choice values are page-relative — no pre-collection). Confirms via `SetGate_*` L-vars, which **lag** — they're polled up to ~6 s (`PollSetGateMatchAsync`), and when changing gates briefly hold the OLD gate. **Every run appends a full walk-log to `%APPDATA%\MSFSBlindAssist\logs\gsx-gate-select.log` — READ THIS FIRST when debugging a "gate not found".** Never choose a `ForbiddenAction` (WARP/Follow-Me/reposition/towing); only choose a positively-identified `SafeServicingAction`. **Leaf matching has a letterless-leaf bare-number fallback**: a GSX menu leaf with no concourse letter ("Parking 209") matches a navdata-lettered target ("P 209" — letter borrowed by the merger for display) on bare number alone; exact identity match is always tried first and the fallback is logged as `MATCH-BARENUMBER` (fixes guaranteed "not found" at EGLL-style airports). **`SelectGateAsync` has an `Interlocked` reentrancy latch** — two Calculate clicks could interleave two DFS traversals on one live GSX menu (the `IsMenuActive` guard reads false during every menu transition, so it cannot prevent this), pressing arbitrary wrong entries; the latch makes the second call return failure immediately.
- **`Services/Gsx/GsxMenuClassifier.cs` is the ONE tunable surface** (pure/stateless; all pattern arrays at the top). Two disambiguations are safety-critical and were each fixed against live menus:
  - **`IsBackUp = IsBack && !IsNext`** (mirror of `IsNextForward`). The back-patterns match "previous"/"◀", which also appear in the pagination entry "◀Previous Page". `BackOutAsync` MUST use `IsBackUp` (the true "↑ Back"), or backing out of a multi-page submenu presses "Previous Page" and the search gets stuck in that submenu (CONFIRMED EDDF). **The "main"/"top" back-patterns are full phrases ("main menu" / "back to top"), not bare words** — bare "main"/"top" false-positived on apron/terminal names like "Main Apron", classifying a drillable category as Back.
  - **Leaf-vs-Category order:** a parking COUNT suffix (`"(N suitable parkings)"`, via `CategoryCountRegex`) is the DEFINITIVE group-header signal → Category. **The count-suffix Category check runs FIRST — before `IsBack` and before leaf matching.** A `"(N suitable parkings)"` group like "Main Apron (12 suitable parkings)" classified as Back made every stand inside unreachable AND desynced `BackOutAsync`; and checking it before leaf matching keeps `"Apron A (8 suitable parkings)"` from being parsed as gate "8". THEN a concrete stand ID → `GateLeaf`, **even when the text contains a structural keyword** like "cargo"/"ramp"/"remote" (so `"Stand F211 … Ramp Cargo"` is a leaf, not a drillable category — CONFIRMED EDDF). THEN structural-keyword-only → Category. Do not reorder.
  - Identity matching: `NormalizeTargetIdentity` / `LooksLikeGate` reduce to concourse+number+suffix (e.g. "F222") and strip a leading 'G' from all-letter navdata names ("GF" → "F").
- **Docking (`Services/DockingGuidanceManager.cs` + `DockingGeometry.cs` + `ProximityBeeper.cs`)** is a standalone state machine (Idle→Armed→Docking→Stopped) fed from the SAME ~30 Hz position handler as taxi guidance but **independent of `TaxiGuidanceManager`**. It owns its OWN `TaxiSteeringTone` instance (lateral cue) + a `ProximityBeeper` (accelerating beep) + spoken metric milestones. To avoid two panning tones, `MainForm` calls `taxiGuidanceManager.SetSteeringToneSuppressed(dockingManager.IsActive)` each frame so only one sounds.
  - `DockingGeometry` is pure static + probe-tested. `alongM` = forward distance to stop, **datum-aligned**: the aircraft DATUM stops at the parking/stop position — an MSFS parking position (and a GSX stop position) is where the aircraft *reference* sits when correctly parked, and the scenery jetway is placed to reach the door for that datum location. **The per-aircraft `gsx.cfg` longitudinal door offset is NOT subtracted from the stop math, and the `SetDoorOffsetMetres` plumbing is removed end-to-end — do NOT reintroduce it.** It describes where the door is ON the airframe, not a stop offset; subtracting it parked a B777 ~26 m short (it had been telemetry-only dead plumbing since that fix). Only the door SIDE cue survives from `gsx.cfg` (see the `GsxAirplaneProfile` bullet). Per-airframe stop depth is the `.py` stop offset's job (`SetStopOffset`), which deice areas keep at `GsxOffset.Zero` (datum-aligned pad). The lateral cue is the **heading-error angle (`hdgErr`)**, NOT `CalculateCrossTrackError` (which assumes the aircraft is ahead of the reference and yields ±180° garbage when docking from behind).
  - **GSX static occupancy clamp (`DockingGeometry.ClampStopToOccupancy`, called in `UpdatePosition` right after `ShiftStop`).** GSX registers an aircraft as PARKED (offers arrival services + operates the jetway) only when its DATUM is inside a circle of radius ≈ the gate's `gatedistancethreshold` centred on `this_parking_pos` (= `_gate.Latitude/Longitude` for GSX gates) — **live-verified at KBOS E13 2026-06-13**: services at ≤25 m incl. 10 m BEHIND the point, reposition past it (so it is a RADIUS, not a forward-only segment; side / `radius{left,right}` do NOT govern static occupancy). The `.py` per-aircraft `SetStopOffset` can push the datum OUTSIDE that circle (KBOS E13 B772: offset → datum 31.5 m > 25 m circle → GSX offered "reposition" after a clean dock — the reported bug). The clamp pulls the post-offset stop back along the gate heading to `gatedistancethreshold − OccupancyClampMarginMetres (2 m)` (FLOORED AT 0 — a malformed profile with `gatedistancethreshold < 2 m` would otherwise pull the datum BEHIND `this_parking_pos`; real thresholds are 15–30 m so this never affects a legitimate gate, probe-locked by the `tiny-threshold` case) **only when** the base stop is itself inside the circle AND the shifted datum would land outside it; it is a **strict NO-OP** for deice pads, navdata-only gates (no `StopLatitude`/threshold), datums already inside the circle, and **VDGS-reliant gates whose stop sits BEYOND the threshold** (EDDF A66: stop 25.1 m, threshold 15 m — those are accepted ONLY via GSX's dynamic VDGS approach-dock, which a static placement even at the exact `.py` target does NOT trigger; clamping them would pull the datum far short of the bar and break jetway reach + the VDGS dock). `ClampBaseStopToleranceMetres (1 m)` keeps the author-set-`gatedistancethreshold == stop-distance` knife-edge (KBOS: both 25) on the CLAMP side despite float/coordinate noise. A one-shot `OCCUPANCY-CLAMP` line goes to `docking-aircraft.log` (latched by `_occupancyClampLogged`, reset per gate in `ResetLocked` — never per-frame IO). Probe-locked (`tools/DockingProbe`: KBOS fires→23 m, EDDF/KATL/inside-circle no-op); a scan of installed GSX profiles found ~850 gates in the affected band across essentially every major airport, so this was a widespread silent failure. Live-verified end-to-end: KBOS E13 (B772 → fixed), EIDW C406 (B738, base stop outside circle → fixed), EIDW C402 (B738, no-op), EDDF A66 (B77W, golden untouched). **Do NOT remove**, do not "simplify" it to clamp on `gatedistancethreshold` unconditionally (breaks EDDF-type gates), and clamp only the `.py`-shifted stop (never the navdata base point).
  - The docking lateral tone uses the **runway-lineup precision profile** (`UpdateHeadingErrorWithThresholds`, 0.5°/1.0°/15°), NOT the width-scaled `UpdateHeadingError` overload (whose `MIN_SCALE` clamp bottoms out at ~1.95°/3.9° — far too loose for parking).
  - `Services/Gsx/GsxAirplaneProfile.cs` reads each aircraft's `gsx.cfg` `[exit<preferredexit>] pos` (col1 = lateral, neg=left), matched by ICAO extracted from SimConnect `ATC MODEL` — **its ONLY consumer cue is the door SIDE** ("Jetway/Door on your left/right" at docking engage); the longitudinal column no longer feeds anything (see the datum-aligned bullet above). The ICAO map is built in the background and refreshed on aircraft change; the build is **single-flight via `Lazy`** (two concurrent multi-second scans no longer race), the `gsx.cfg` scan is **depth-bounded (6)** so texture/sound trees are never crawled, and `UserCfg.opt` parsing **excludes `InstalledPackagesPathNextBoot` lines** (which could resolve a not-yet-active packages path after a relocation). `MainForm.OnAircraftIcaoTypeDetected` locks `_refreshedIcaos` (concurrent `Task.Run` mutations of a plain HashSet) and rechecks the ICAO is still current before publishing the door side, so a late Refresh for the OLD aircraft can't clobber the NEW one.
- **Docking completion stops taxi guidance.** `DockingGuidanceManager` raises a `DockingCompleted` event once on the Docking→Stopped transition (including overshoot), fired OUTSIDE its lock. `MainForm` subscribes and calls `taxiGuidanceManager.StopGuidance()` (thread-safe + silent) so the flow ends cleanly instead of taxi sitting in LiningUp forever after parking.
- **Arrival ownership is ENGAGE-LATCHED, not gate-set.** `MainForm` feeds `taxiGuidanceManager.SetDockingActive(dockingGuidanceManager.IsActive)` — `IsActive` = the docking state machine is in Docking or Stopped, exposed as a **lock-free volatile snapshot** (one fewer docking-lock + zero SettingsManager static-lock reads per 30 Hz frame). Taxi speaks its FULL arrival sequence — parking countdown, "Stop. Hold position.", "Align with X", "Destination reached", "Parking brake." — right up until docking actually engages; once engaged, docking owns the rest of the arrival through Stopped and taxi stays quiet. **Do NOT widen this back to gate-set semantics** (the old `OwnsArrival` = gate set + docking enabled): a navdata gate where docking never engages (approach outside the 70° cone, stop beyond engage range, approximate navdata heading) then arrives in TOTAL verbal silence. The deliberate trade: a brief sequential overlap is possible (taxi's stop callout fires, then docking engages with its own countdown) — that's self-correcting and far better than silent arrivals. The same `IsActive` read also drives the steering-tone mute via `SetSteeringToneSuppressed`. **PENDING-aware wording (third state between "not engaged" and "engaged"):** `DockingGuidanceManager.IsArmedAwaitingEngage` (volatile snapshot, set per-frame in the Idle/Armed branch when detail math is available, the stop resolves >3 m ahead, and engage conditions aren't met; cleared on engage/reset/dispose) feeds `taxiGuidanceManager.SetDockingPending`. While pending, taxi's arrival wording REDIRECTS FORWARD — align announce gains "Then continue ahead for docking guidance.", aligned-enter says "Continue ahead. Docking guidance will take over." (NOT "Parking brake."), the 10 ft countdown directive and stopped-in-zone cue say continue instead of "Stop. Hold position.", and the LiningUp status readout says continue. Why: the GSX stop is routinely tens of metres past the navdata point where taxi's route ends, and a gate's GSX `gatedistancethreshold` can shrink the engage range BELOW that gap — KATL F3 2026-06-11: pilot was told "Parking brake." at the navdata point and sat 26 s at 33.7 m with docking Armed (engage range ~24 m from the gate's threshold), until they crept forward on their own. Announcements only — engage logic, tone ownership, and the engage-latch trade above are unchanged; a pending gate where docking never engages still gets the full (redirected) verbal arrival.
- **Jetway-precise lateral lineup (`DockingGuidanceManager.ComputeLineupError`).** Intercept dead-band tightened **8 ft → 1 ft** (the 8 ft band stopped correcting cross-track below 8 ft, parking up to ~2.4 m off centerline); `SaturationFt` 60 → 40 (a small residual still yields a usable correction angle). Cross-track convergence is a function of DISTANCE travelled, not time (`d(cross)/d(forward) = −sin(angle)`), so it closes the same per metre at 1 kt as at 5 kt — no slow-speed special-casing — and the continuous sqrt ramp never springs a late turn.
- **Final alignment turn completes earlier.** The intercept-fade squares the heading to pure gate heading by **2.5 m out** (`FadeStartM`/`FadeEndM` = 6/2.5, was 4/1), so an over-rotated gate entry (~5° off the taxi turn) finishes the squaring turn with room to creep straight in instead of cramming it into the final metre and stopping ~2° off.
- **Precise, unforgiving stop + persistent "docked" tone.** `StopToleranceMetres` tightened **0.5 → 0.3 m** (drives gate IsStop, the solid tone, and the "GSX docking complete." callout), so the pilot lands within ~0.3 m of the exact stop. The beep plateau is removed: `BeepNearMetres = StopToleranceMetres`, so the accelerating pulse keeps speeding up right to the stop with NO max-speed plateau — the old 2 m plateau made everything from 2 m to the stop sound identical, so pilots read "fast beep" as "stop" and parked short, mid-turn. Now: accelerating pulse = keep creeping; solid tone = stop here. The SOLID continuous tone was previously DEAD CODE — the state machine called `_beeper.Stop()` at the same 0.3 m threshold the solid tone begins, so the beep just vanished at the stop. Fixed: at IsStop the lateral pan tone stops but the beeper is held in its solid mode and KEEPS sounding through the Stopped state as a "docked — hold position" marker, until the pilot ends guidance (Stop taxi guidance button → `SetDestinationGate(null)` → `ResetLocked` stops it) or taxis/backs away (see the lifecycle bullet below). An OVERSHOOT stop does NOT hold the solid tone — a "docked" marker over a bad park misleads — and no longer disposes the beeper, so a back-up-and-retry dock re-engages with working audio.
- **Docking state lifecycle — Stopped is escapable, every state is exitable.** (a) **Taxi-away disengage uses ABSOLUTE distance, not along-track**: along-track goes NEGATIVE once the stop is behind the aircraft, so the old `alongM > 75` check could never fire for a forward taxi-out and the state (and solid tone) latched forever — raw distance >75 m disengages in ANY direction, including a stale next-flight gate hundreds of km away. (b) **Backing up >3 m past the stop re-arms to Idle** (`RearmBackupMetres`) so a back-up-and-retry dock re-engages through the normal `ShouldEngage` path with fresh audio + milestones. (c) **Disabling docking (or losing the gate) mid-approach fully resets** (`ResetLocked`, not just silence) — leaving `_state` latched at Docking/Stopped kept `IsActive` true forever, muting taxi's steering tone with no lateral cue for the final gate turn. (d) **Stopped-short closure**: engaged + sitting still ≥4 s with 0.3–10 m to go → *"X to stop. Continue forward."* — taxi's stopped-in-zone "Stop. Hold position." cue is suppressed while docking owns the arrival, so docking must provide its own closure instead of an endless fast-but-not-solid beep. (e) **Stale-gate lifecycle**: takeoff-assist activation and `LandingRollout` entry both call `SetDestinationGate(null)` — a stale departure gate could otherwise keep `IsActive` latched on landing and mute the landing-exit rollout steering tone.
- **Gate-lineup "aligned" band is docking-aware; gate lineup NEVER pulses.** Tight (1° / 12.5 ft enter, 2° / 25 ft exit) ONLY while docking is engaged (`_dockingActive` — docking's tone + 0.3 m stop own the precision and taxi's verbal is suppressed anyway); forgiving (4° / 25 ft enter, 7° / 40 ft exit) otherwise, because the navdata parking point is routinely metres off the real stand and demanding ~12 ft to it left correctly-parked pilots permanently "not aligned" (no "Parking brake." cue). The runway-style stopped-misaligned pulse was briefly enabled for gate lineup and is REMOVED — precision parking is docking's job, and pulsing 3 Hz at a correctly-parked pilot demanding precision to a possibly-offset navdata point is a misfeature. Runway lineup keeps its pulse (its centerline reference is authoritative). Do not re-add a gate pulse.
- **Hot-path perf invariants (do not regress):** docking far-field telemetry + lineup math are gated to `<150 m` raw distance or engaged (`DetailRangeMetres`) — no more 2/s `docking.log` file appends on the SimConnect thread, under the docking lock, for the whole taxi; hold-short label formatting happens only in firing branches and the hold-short/parking/exit-approach/runway-end callout paths early-out once all their latches have fired (don't reintroduce per-frame milestone-table builds); `TaxiAssistForm`'s gate list is cached per ICAO as (spot, resolved node) pairs and the search box / fit filter filter IN MEMORY per keystroke (don't reintroduce per-keystroke `GetGates` + navdata query + nearest-node resolution on the UI thread — matches the `GateTeleportForm` pattern); `SettingsManager.Save` serializes under the static lock but writes the file OUTSIDE it (don't move the IO back under the lock — an options-dialog save mid-approach stalled the 30 Hz position path behind an antivirus-scanned `%APPDATA%` write).
- **`docking.log` telemetry includes absolute coords.** Lines now carry `stopLat`/`stopLon` (computed stop target) and `acLat`/`acLon` (aircraft position) alongside the relative `along`/`crossFt`, so docking's target can be compared directly against a known-correct GSX position.
- **SimConnect heading-unit gotcha (live verification).** SimConnect's `PLANE_HEADING_DEGREES_TRUE` / `_MAGNETIC` are returned in **RADIANS**, not degrees, despite the name — multiply by 57.2958 (180/π) to get degrees (a logged 5.93 = 339.7°). Latitude/Longitude ARE in degrees. Applies to any future MCP/SimConnect live-verification of docking.
- **GSX operational note: engines OFF for services.** GSX will not offer ground services (deboarding/jetway/etc.) while engines are running — it prompts "stop engines to request services." This is GSX behaviour, not an MSFSBA bug; a precise dock with engines running still won't show service options.
- **Verified live (EDDF A66 / B77W, engines off):** offset 5.30 m applied; docking target (50.04691716, 8.56034700) heading 339.9° true; a real dock stopped ~0.6 m short + 0.27 m left of centerline and GSX accepted services — confirming the target is GSX-correct.
- **Distance units (`Settings/DistanceUnit.cs`, `Services/DistanceFormatter.cs`, `Services/DistanceMilestones.cs`).** Default metres, optional feet. `DistanceFormatter` is the DISPLAY layer only — its `UnitProvider` is wired to `SettingsManager.Current.GroundDistanceUnit` at startup (`MainForm`); **never use it for guidance thresholds** (those stay unit-native internally). `DistanceMilestones` returns **unit-native** countdown tables (round numbers in the active unit, e.g. docking 30/20/10/5 m vs 100/60/30/15 ft — NOT literal conversions); triggers stay metric so guidance logic is unaffected. **`GroundTrafficUseMetres` is a SEPARATE, independent toggle** — do not fold it into `GroundDistanceUnit`. **Long taxi distances (>~6000 ft, `TaxiGuidanceManager.FormatDistance`) switch to the big unit MATCHING the setting** — kilometres in metres mode, nautical miles in feet mode; a metric user never hears NM for a ground distance.
