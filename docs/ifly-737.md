# iFly 737 MAX8

Support for the iFly 737 MAX8 (`IFLY_737MAX8`) via the **official iFly SDK** — shared-memory reads + WM_COPYDATA writes. No MobiFlight, no L:var writes.

`Aircraft/IFly737MAXDefinition*.cs` (`IFLY_737MAX8`, 5 partials) + `SimConnect/IFly/` + `Forms/IFly737/`. **The transport is the OFFICIAL iFly SDK, not L:vars/MobiFlight** (the 2026-03 FLYINGTOMO-addon design is SUPERSEDED — do not resurrect it): READS poll the shared-memory block `iFly737MAX_SDK_FileMappingObject` (`IFlySdkClient`, 250 ms, v1.5 mutex when present); WRITES send WM_COPYDATA command messages (`iFly737MAX_MSG_GAU`, `{int Command; double V1,V2,V3}`, **cbData = sizeof+2 exactly** — the SDK sample's trailing 2 bytes are load-bearing) to the "iFly Plugin - MSFS2024" / "iFly Plugin" window. Fully independent of SimConnect; works with no extra add-ons. Key rules:
- **`IFlySdkOffsets`/`IFlySdkFields` are GENERATED** (`tools/ifly-gen/generate_ifly_sdk.py` from SDK_Defines.h) — regenerate, never hand-edit. Field names in the def are the SDK struct names (arrays flattened `Name_i`, 0 = Captain/left). The def registers **521 variable keys** (`_vars`, the live `IFly737MAXDefinition.GetVariables()` count) — all audit-verified against the table; the SDK's own flattened struct table is larger, 690 fields (`grep -c "new(" IFlySdkFields.cs`), since per-digit fields consumed only by the `SYN_*` synthetic composer (below) and other unexposed struct fields never get an individual `_vars` entry. Vars are declared `Type = PMDGVar` so they're excluded from every SimConnect batch; MainForm bridges `Sdk.VariableChanged` → `OnSimVarUpdated` (`OnIFlyVariableChanged`, the PMDG CDA event shape, `IsInitialSnapshot` seeds silently).
- **Synthetic `SYN_*` fields**: the MCP windows / transponder code / RTP frequencies / ELEC LED / IRS display / fuel gauges are stored PER-DIGIT in the SDK — `IFlySdkClient.RaiseSyntheticEvents` composes each window per poll and fires ONE `SYN_*` event on change (numeric value, blank = −99999; text displays carry a content hash and render from the live snapshot via `TryGetDisplayOverride`). Never key a definition on a single digit field. Seven of these (`SYN_MCP_SPEED/HEADING/ALTITUDE/VS/COURSE_1/COURSE_2`, `SYN_XPDR_CODE`) are registered `announced: true` and so are also Ctrl+M-listable, same as any other announced var.
- **Command-encoding traps are documented inline in the partials** — the SET commands' Value2 encodings frequently DIFFER from the status field (engine start levers are INVERTED 0=IDLE/1=CUTOFF; FO display selector is reversed vs Captain; position lights reversed; landing lights 0/1 status vs 0/1/2 command; guarded switches use the 1-based `*_Mode` companion field with `map: v => v-1` + `Value3 = 1` guard-bypass). Check the comment before "fixing" a map lambda.
- **No absolute SET exists for**: squawk (keyed digit-by-digit via `FMS_XPNDR_KEYPAD_*`, 120 ms paced), RTP standby frequencies (WHOLE/FRACT INC-DEC rotaries stepped to target, 40 ms paced, read back from the display), and baro VALUE (Ctrl+B dialog uses stock `KOHLSMAN_SET` — the aircraft tracks the stock Kohlsman).
- **Staleness watchdog** (`IFlySdkClient`): a named shared-memory section survives while ANY handle is open, so a sim/plugin exit alone doesn't fail the read — `Tick18` (advances every FS tick while the plugin is alive) is checked each 250 ms poll instead; 12 consecutive polls with no movement (~3 s) flags the connection stale, firing `ConnectionChanged(false)` and clearing `IsReady` — a PAUSED sim also reads as stale, by design. On resume the client re-seeds silently (initial-snapshot semantics), never replaying the gap as announcements.
- **Per-send plugin-window resolution** (`SendCommand`): the WM_COPYDATA target HWND is re-resolved via `FindWindowEx` on EVERY send, never cached — a cached handle can be recycled by Windows after a sim restart, and a send to the recycled window "succeeds" (delivered) while the aircraft hears nothing; the title scan costs microseconds against 40-120 ms-paced commands. Send-failure logging is edge-triggered (`_lastSendFailed`) so a sustained-failure loop logs once, not once per send.
- **Announce logic** (`ProcessSimVarUpdate`): annunciators announce lit-edge only (dim + bright both = lit) through a flash filter (`HandleLightEdge`/`OnOffSweepTick` — "on" once at the lit edge, "off" only after the light has stayed off 2 s), suppressed while the master LIGHTS TEST is held (`Lights_Test_Status == 0`); the A/P and A/T disengage warning lights (`AP_Indicators_Light_Status_{0,1}`, `AT_Indicators_Light_Status_{0,1}`) blink until reset and use the same filter, speaking their color (amber or red); MCP mode buttons and warning push lights decode the 0-5 switch+light encoding (`value % 3 > 0` = lit). Because these announce from INSIDE ProcessSimVarUpdate, **the Ctrl+M monitor manager mute needs the HS787-style Suppressed-wrap** — wired in MainForm Step 2.5 (`iflyMuted` → `IFlyDisabledMonitorVariables`) alongside the generic Step-6 gate for the plain switch combos; the off-sweep timer runs OUTSIDE that wrap, so it checks `IFlyDisabledMonitorVariablesSet` itself before speaking a held-off light. The autopilot window's own writes are separately echo-suppressed (`WindowEchoActive`, consulted in `HandleLightEdge`'s on-path, `OnOffSweepTick`'s off-path, and MainForm's Step-6 gate) so a button click doesn't re-announce the state NVDA already read off the renamed button.
- **LIVE-VERIFY** (unconfirmed against a live sim — flagged inline where relevant, don't "fix" without re-testing): `BARO_STD_Status` latch-vs-momentary (the Ctrl+B "Standard" toggle's state source — unknown whether the SDK field is a true latch or reflects a momentary command); ND range positions 3-10 (the `ND_Range_Status` struct comment says `0~2`, but the def follows the `RANGE_SET` command doc's `0..10` as authoritative — see the ENCODING TRAP comment in the ForwardPedestal partial); RTP standby frequency stepping uses 25 kHz `WHOLE/FRACT INC/DEC` steps — unverified against 8.33 kHz-spaced channels; the IRS display's inferred decimal-point positions (left after L2/L3/L4, right after R3/R4/R5, from `IFlySdkSnapshot`'s field append order) are untested against a live IRS window; `MA_1`/`MA_2` master-annunciator identity is unknown and the fields are deliberately NOT registered (see the "LIVE-VERIFY before ever re-adding" comment near `AT_Light_Status` in the core partial); and Ctrl+B's `KOHLSMAN_SET` write carries no altimeter index, so whether the iFly tracks ONE Kohlsman for both sides or just the Captain's (the dialog's "applies to both altimeters" claim) is unconfirmed.
