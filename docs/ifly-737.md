# iFly 737 MAX8

Support for the iFly 737 MAX8 (`IFLY_737MAX8`) via the **official iFly SDK** — shared-memory reads + WM_COPYDATA writes. No MobiFlight, no L:var writes.

`Aircraft/IFly737MAXDefinition*.cs` (`IFLY_737MAX8`, 5 partials) + `SimConnect/IFly/` + `Forms/IFly737/`. **The transport is the OFFICIAL iFly SDK, not L:vars/MobiFlight** (the 2026-03 FLYINGTOMO-addon design is SUPERSEDED — do not resurrect it): READS poll the shared-memory block `iFly737MAX_SDK_FileMappingObject` (`IFlySdkClient`, 250 ms, v1.5 mutex when present); WRITES send WM_COPYDATA command messages (`iFly737MAX_MSG_GAU`, `{int Command; double V1,V2,V3}`, **cbData = sizeof+2 exactly** — the SDK sample's trailing 2 bytes are load-bearing) to the "iFly Plugin - MSFS2024" / "iFly Plugin" window. Fully independent of SimConnect; works with no extra add-ons. Key rules:
- **`IFlySdkOffsets`/`IFlySdkFields` are GENERATED** (`tools/ifly-gen/generate_ifly_sdk.py` from SDK_Defines.h) — regenerate, never hand-edit. Field names in the def are the SDK struct names (arrays flattened `Name_i`, 0 = Captain/left); all 346 registered bindings are audit-verified against the table. Vars are declared `Type = PMDGVar` so they're excluded from every SimConnect batch; MainForm bridges `Sdk.VariableChanged` → `OnSimVarUpdated` (`OnIFlyVariableChanged`, the PMDG CDA event shape, `IsInitialSnapshot` seeds silently).
- **Synthetic `SYN_*` fields**: the MCP windows / transponder code / RTP frequencies / ELEC LED / IRS display / fuel gauges are stored PER-DIGIT in the SDK — `IFlySdkClient.RaiseSyntheticEvents` composes each window per poll and fires ONE `SYN_*` event on change (numeric value, blank = −99999; text displays carry a content hash and render from the live snapshot via `TryGetDisplayOverride`). Never key a definition on a single digit field.
- **Command-encoding traps are documented inline in the partials** — the SET commands' Value2 encodings frequently DIFFER from the status field (engine start levers are INVERTED 0=IDLE/1=CUTOFF; FO display selector is reversed vs Captain; position lights reversed; landing lights 0/1 status vs 0/1/2 command; guarded switches use the 1-based `*_Mode` companion field with `map: v => v-1` + `Value3 = 1` guard-bypass). Check the comment before "fixing" a map lambda.
- **No absolute SET exists for**: squawk (keyed digit-by-digit via `FMS_XPNDR_KEYPAD_*`, 120 ms paced), RTP standby frequencies (WHOLE/FRACT INC-DEC rotaries stepped to target, 40 ms paced, read back from the display), and baro VALUE (Ctrl+B dialog uses stock `KOHLSMAN_SET` — the aircraft tracks the stock Kohlsman).
- **Announce logic** (`ProcessSimVarUpdate`): annunciators announce lit-edge only (dim + bright both = lit), suppressed while the master LIGHTS TEST is held (`Lights_Test_Status == 0`); MCP mode buttons and warning push lights decode the 0-5 switch+light encoding (`value % 3 > 0` = lit). Because these announce from INSIDE ProcessSimVarUpdate, **the Ctrl+M monitor manager mute needs the HS787-style Suppressed-wrap** — wired in MainForm Step 2.5 (`iflyMuted` → `IFlyDisabledMonitorVariables`) alongside the generic Step-6 gate for the plain switch combos.
- **Windows**: Shift+M = `IFly737CDUForm` (renders the SDK's 14×24 character-cell CDU screens, Left/Right selectable, 80 ms-paced key queue via `FMS_CDU_{1,2}_*` commands); Shift+T = `IFlyEfbForm` (the SP1 iPad-style EFB served over HTTP by iFly's own EFB process, hosted in WebView2 — port from `UserSettings.IFlyEfbPort`, default 8084, must match iFly Manager). **EFB accessibility LIVE-VERIFIED WITHOUT THE SIM (2026-07-15):** `Data\Tool\EFB\iFly-EFB.exe` (Qt6, standalone) runs and serves on 8084 with no sim; the pages are REAL semantic HTML (Performance module alone: 29 `<input>`/17 `<button>`/20 `<select>`/52 `<label>`, ZERO canvas; balance/doors/groundservices/myplane/failures all semantic) — modules are JS views loaded into `#app` via `loadJavaScriptModule('/js/<m>/<m>.js')`, live data over socket.io. The ONLY non-semantic module is `cdu` (irrelevant — MSFSBA's Shift+M FMC covers it, and the EFB HIDES its CDU icon for browser/efb1-efb2 clients); the in-EFB Manuals are page PHOTOS (point users at the real PDFs in `Data\Documentation`). `IFlyEfbForm.AccessibilityShim` (injected on every NavigationCompleted; **shim itself END-TO-END TESTED against the live EFB via a scratch WebView2 harness, all checks green 2026-07-15**) fixes the real hurdles: auto-dismisses the UNLABELED full-screen `#lockScreenOverlay` ("click to unlock", created on EVERY load — `window.unlockScreen()` + node removal, retried ~5 s because it spawns async); stamps `role=button`+`tabindex` on the `.app-icon` clickable divs, the `#homeButton` (title-only div — labeled "Home"; in-page **Alt+Home** also clicks it) and the Ground Services `.ground-support-category-item` sidebar tabs (+ Enter/Space → click on any stamped div, MutationObserver re-applies across SPA view swaps); English `aria-label`s on the Doors page's `button[data-door]` (per-door buttons are textless; the arm buttons' only name was a CHINESE tooltip 预位/解除预位); and **resurrects the hidden Ground Vehicles panel** — GPU/air-start/AC-cart/fuel/lavatory/water/airstairs/catering/cargo are toggled by clicking the graphical SVG aircraft diagram, but every vehicle has a REAL sim-wired checkbox the module keeps hidden ("功能已集成到SVG", kept for data binding; the SVG click just toggles it) → the shim unhides + labels all 12 and injects a "Ground Vehicles" category tab (live-verified: tab activates, 12 labeled checkboxes visible, Home returns to the app grid). ⚠ the SVG diagram reuses ids like `electric` — resolve vehicle checkboxes with `input[type=checkbox][id=...]`, never bare getElementById; Ctrl+M = `IFly737MonitorManagerForm`; Ctrl+S/H/A/V/B = MCP/baro `ValueInputForm` dialogs (PMDG 737 shape); Ctrl+P = `IFly737AutopilotWindow` (Salty-747 shape, def-cached + disposed in `Shutdown()` — engage cluster: CMD A/B, CWS A/B, FDs, A/T arm, disengage bar, momentary A/P + A/T disconnects; live-state button labels off the snapshot). FMS V-speeds/perf come from the iFly WASM's plain L:vars (`iFly737MAX_Lvar_*_VAL`, FMS Data panel). Checklist: `Checklists/iFly_737MAX8_Checklist.txt`. On aircraft swap `SwitchAircraft` calls `oldIFly.Shutdown()` (disposes the poll + mapping) and `DisposeIFlyForms()`.
- **Readout hotkeys (PMDG-737 parity, 2026-07-15 finish pass)**: F/Shift+F = total fuel lbs/kg and W/Shift+W = gross weight lbs/kg (stock `FUEL TOTAL QUANTITY WEIGHT`/`TOTAL WEIGHT` via `RequestSingleValue` — unit-exact regardless of the cockpit gauge unit option; per-tank gauge digits stay on the Fuel panel); L = flaps (SDK `FLAP_Status` detent 0-8 → UP..40 + "in transit" from the TE flap lights); Shift+G = gear (SDK lever + per-gear red/green lights → locked down / in transit / up); B = altimeter (stock Kohlsman `ALTIMETER_SETTING` SimVar, registered Continuous+IsAnnounced with a deduped "Altimeter: hpa, in.hg"/"standard" announce on knob change — the iFly tracks the stock Kohlsman, which is also how Ctrl+B sets it). **The FMS Data panel is display-only and MUST keep its empty `PanelList(P)` entry** — without a panel-controls entry MainForm's panel builder renders NOTHING (the HS787 Flight-Data-panels bug). **Read-only status combos (`Sw(..., set: null)` — Lights Test position, Ground Service, CVR switch, Minimums Reference, Baro Units, fire-switch states, flap/slat lights) are caught in `HandleUIVariableSet`**: announce "read-only indicator" + `Sdk.RefireAllFields()` (initial-snapshot events update combos silently) so the combo snaps back to truth instead of latching a selection the aircraft never took. `TryGetDisplayOverride` also renders the `SYN_RTP{1-3}_ACTIVE/STANDBY` frequency windows ("Blank" when unpowered — never the raw −99999 sentinel) and the `FUEL_TEMP_Indicator` ≤ −100 invalid sentinel as "Not available".
