# Gemini AI

> **Read this when:** working on the Gemini AI features (display reading, scene/route description, model selection, retry/backoff).

## Gemini AI (display reading + scene/route description)

**AI provider abstraction (`Services/IAiProvider.cs`).** The three AI features (on-demand cockpit-display reading, scene description, the Shift+E route briefing) run behind `IAiProvider` — `AnalyzeDisplayAsync` / `AnalyzeSceneAsync` / `DescribeRouteAsync` / `ListAvailableModelsAsync`. Two implementations: **`GeminiService`** (Google) and **`ClaudeService`** (Anthropic). The user picks one in the unified Settings dialog's **AI panel** (`Forms/Settings/AiSettingsPanel.cs`, a provider combo Gemini/Claude + each backend's fields; model lists populate LAZILY — only the visible provider's list is fetched on dialog open, the other on first switch, so opening Settings never fires two network calls), persisted as `UserSettings.AiProvider`; **`Services/AiProviderFactory.Create()`** returns the chosen backend. The three call sites — `MainForm.DescribeSceneAsync`, `BaseAircraftDefinition.ReadDisplay`, `ElectronicFlightBagForm` route — build via the factory and never branch on provider. **Hard switch: when Claude is selected ALL three features go through Claude; no silent cross-provider fallback.** Shared bits: `AiModelInfo` (the dropdown item, `ToString()` is what NVDA speaks) and the three provider-neutral prompt builders (`GeminiService.Get*Prompt`, now `internal static`, reused by `ClaudeService` so wording stays in sync). `DisplayType` stays nested on `GeminiService` (the five aircraft defs reference `GeminiService.DisplayType` — don't churn them).

**Claude (`Services/ClaudeService.cs`).** Raw `HttpClient` + Newtonsoft (mirrors the Gemini transport, NO Anthropic SDK dependency). `POST /v1/messages` with `x-api-key` + `anthropic-version: 2023-06-01` headers set PER-REQUEST (the HttpClient is static/shared); vision = base64 `image` content block before the text block; route briefing adds the `web_search_20260209` server tool when `UserSettings.ClaudeWebSearch` is on (the NOTAM-grounding equivalent of Gemini search grounding — extra cost; this is the current dynamic-filtering variant, supported on the default Opus 4.8 + Sonnet 4.6. If the selected model rejects it — e.g. Haiku 4.5 — `DescribeRouteAsync` catches the web_search-named 400, retries ungrounded, and PREFIXES the briefing with a spoken "not grounded with current NOTAM or weather data" note — the degrade must stay AUDIBLE, never silent, per the no-silent-fallback rule), **capped at `max_uses=5`** (an uncapped multi-airport NOTAM briefing runs 10+ searches → slow + $10/1000 searches). **Web-search "process narration" stripping:** the route call also sends a `system` prompt telling the model to output only the briefing, AND `ParseResponse` keeps only the text AFTER the last tool block (`server_tool_use`/`web_search_tool_result`) — together these drop the "Let me search for NOTAMs… now I'll compose the briefing." narration that otherwise leaked ahead of the briefing (a plain vision/text response has no tool blocks, so all text is kept). **`ParseResponse` also surfaces `stop_reason`:** on `max_tokens`/`pause_turn` it appends "(Response may be incomplete — Claude stopped before finishing.)" — or a clear stopped-early message when no answer text survived — because a blind user cannot visually notice a briefing that just stops. **`max_tokens` is REQUIRED by the Messages API** (cannot be omitted); it is 16000 — effectively uncapped for these outputs (a briefing is ~700 tokens) and the documented ceiling for non-streaming requests (larger values need SSE streaming). **Vision screenshots are downscaled client-side to ≤2576 px long edge** (`ComputeScaledSize` + a System.Drawing re-encode; failures fall back to the original bytes): Anthropic caps each image at 5 MB (a 4K PNG screenshot can exceed it) and current models accept at most 2576 px on the long edge before downscaling server-side anyway, so nothing is lost — 1080p/1440p pass through untouched. The pure logic (narration strip, stop_reason, retry delays, scaled sizing) is pinned by `tests/MSFSBlindAssist.Tests/ClaudeServiceTests.cs`. Models from `GET /v1/models` (newest-first, filtered to `claude-*`). Same retry (429/5xx/timeout/connection) + fail-fast (400/401/403/404) shape as Gemini. **No thinking param** (omitted → no extended thinking on Opus 4.7/4.8; keeps display reads fast + cheap). **User-supplied key only** (`UserSettings.ClaudeApiKey`); MSFSBA never pays centrally. Default model `claude-opus-4-8` (Anthropic has no `*-latest` alias — the dropdown lets the user pick Sonnet/Haiku for lower cost). Cost rule of thumb: ~1.5¢/display read on Opus 4.8, so ~$5/month covers a typical hobby flyer; Haiku 4.5 is ~5× cheaper.

**Live-verified (2026-06, real Anthropic API):** Opus 4.8 text + vision work end-to-end — it read a real 1920×1080 MSFS cockpit screenshot exactly (PFD ALT/QNH/heading/THS, ND mode/waypoint/TCAS). **Fable 5 (`claude-fable-5`) is LISTED by `GET /v1/models` but 404s on use** for non-enrolled accounts ("Claude Fable 5 is not available. Please use Opus 4.8" — Fable/Mythos are access-gated programs). So it appears in the AI tab's model dropdown but, if selected, the 404 fail-fast handler shows "model 'claude-fable-5' is unavailable — choose a different model" (graceful, no crash). Don't special-case it; the list endpoint reports models you can *see* but may not be entitled to *use*. **web_search caveats (live Anthropic docs):** dynamic filtering (more accurate, fewer tokens) **requires also declaring the `code_execution` tool** — MSFSBA declares only `web_search`, so it currently gets BASIC search; NOTAM coverage via web search is inherently weak (NOTAMs live in FAA/ICAO systems, not well web-indexed — a dedicated NOTAM API is the real fix, not tool tuning). If web-grounded briefings ever exceed the 120s `HttpClient` timeout, **streaming** is the proper fix (SSE keeps the connection alive through the search pause); `max_uses=5` is the current lighter mitigation. **⚠️ Never paste an API key into chat/logs/code — user keys live only in `UserSettings.ClaudeApiKey` (entered in the Settings dialog's AI tab).**

`Services/GeminiService.cs` powers the Gemini backend (on-demand cockpit-display reading, scene description, and the Shift+E route briefing). Key points:
- **Model is USER-SELECTABLE, not a fallback chain.** The model used for every AI call is `UserSettings.GeminiModel` (default `gemini-flash-latest`, a rolling alias). The AI settings panel (`Forms/Settings/AiSettingsPanel.cs`, Gemini section) populates a `DropDownList` from a LIVE fetch — `GeminiService.ListAvailableModelsAsync()` (`GET …/v1beta/models`, filtered to `generateContent`-capable text/vision models, newest-first) — with a curated fallback list (`gemini-flash-latest`, `gemini-3.5-flash`, `gemini-2.5-flash`) when offline / no key / fetch fails. Do NOT reinstate a silent multi-model fallback: it hid which model produced a response, which the user rejected.
- **Do NOT send `thinkingConfig`/`thinkingBudget`.** `thinkingBudget` is the Gemini 2.5 parameter; Gemini 3.x models use `thinking_level` and treat `thinkingBudget` as deprecated (and error if both are set). The default models (`gemini-3.5-flash`, `gemini-2.5-flash`) already enable dynamic thinking by default, so omitting `thinkingConfig` keeps thinking on for accuracy across every selectable model without sending a wrong/redundant parameter.
- **Reliability = per-request retry/backoff.** `SendRequestAsync` retries transient failures (429, 500/502/503/504, HTTP timeout, connection-level `HttpRequestException`) with backoff and fails fast on client errors (400/401/403/404, where 404 means "model unavailable — pick another"); on exhaustion it throws a clear "Gemini is busy or unavailable — please try again" message. The timeout catch must NOT gate on `ex.CancellationToken.IsCancellationRequested` (false on a modern .NET HttpClient timeout) — catch `TaskCanceledException` as the timeout (the `SimBriefService` pattern).
