# PMDG EFB (Coherent debugger)

> **Read this when:** working on the PMDG 737/777 EFB tablet transport (Coherent GT debugger, scrape agent, noise-suppression rules).

### PMDG EFB (Coherent debugger)

The PMDG EFB tablet is read and driven over the MSFS Coherent GT remote debugger (`http://127.0.0.1:19999`), like the FBW A380 flyPad. NO Community mod package, NO HTTP bridge, NO HTML patching, and NO sim restart are required.

**Architecture:** `SimConnect/CoherentPmdgEfbClient.cs` (implements `IMcduBridge`) connects to the `PMDGTablet` Coherent view, installs the generic in-page agent `Resources/coherent-pmdg-efb-agent.js` (ES5: resolves controls by TYPE not element id, so new PMDG pages read automatically), and polls `scrape()` — capturing readable document TEXT (linearizing the SimBrief OFP `<pre>` briefing + NOTAMs), two-column label↔control pairing, checkbox active-unit from the EFB `Settings` object, fa-icon labeling, leaflet-marker skip, and duplicate-button id disambiguation. Transport: `CoherentPmdgEfbClient` exposes `click_mcdu_element(idx)` / `set_element_value(idx, text)` commands (same shape as FBW A320/A380). UI: reuses `Forms/FBWA380/FbwEfbForm.cs` WebView2 browse view (the same form the FBW flyPad uses), with a monospace `<pre>` render branch for the OFP. On startup, `Patching/LegacyEfbBridgeCleanup.RemoveRetiredBridges()` (called from `Program.Main`, best-effort, never throws, no-op once gone) removes the retired Community-folder accessibility bridges — the PMDG `zzz-pmdg-efb-accessibility` package (via `EFBModPackageManager.Remove`) AND the HorizonSim 787 bridge (via `Patching/Hs787LegacyUninstaller.Remove`, which reverts BOTH the FS2020 `zzz-hs787-accessibility` override package and the FS2024 in-place HTML patch — the latter restored from its `.msfsba_backup` copies). `Hs787LegacyUninstaller` is the removal half of the deleted `HS787ModPackageManager` (uninstall only — the installer was intentionally dropped in the 787 Coherent migration). Both bridges are superseded by the Coherent-debugger transport.

**Noise-suppression / label-robustness rules in `collect()` (all live-validated 2026-06-20 against the 777 tablet, jsdom-locked in `tools/pmdg-efb-test`):**
- **Drop truly-anonymous buttons.** A button whose label resolves empty (no text, no fa-icon, no title/aria/id, no row-pair) is NOT emitted — after a SimBrief/weather load the Dashboard map sprouts unlabeled leaflet overlay/control buttons the `leaflet-marker-icon` skip misses; they'd read as a flood of "(button)". A button with an id still gets its id-derived name (so this only drops genuinely nameless ones).
- **Drop single-character glyph text.** Standalone text-content whose trimmed length ≤ 1 is a decorative graphic glyph, never content — the Take-Off page renders the runway designators "35R"/"17L" as per-character `<span>`s ("3","5","R","1","7","L") inside `#opt_takeoff_runway.runway_container`. Skipped BEFORE the same-row merge so "R"+"1" can't combine into "R 1". The real value still reads from the Runway ID dropdown.
- **A field label must contain letters.** `_rowLabel` skips value-DISPLAY labels (the weather widget renders Temp/Wind/Dewpoint/QNH as `<label class="pmdg_measurement">28</label>` — a value, not a field label) and requires the matched label to contain a letter; the two-column pairing applies the same letter-guard. Without this the "Toggle Weather" checkbox + "Weather Icao" field borrowed the METAR temperature ("28"/"31"). The id fallback then yields the correct "Toggle Weather"/"Weather Icao".
- **Icon-button collisions read clean id names.** When two icon-only buttons collide on a generic fa glyph (after a weather load BOTH the status-bar refresh and the weather-search button show fa "Refresh"), the disambiguation uses the descriptive id name AS the label ("Statusbar Restart" / "Weather Search") instead of the clunky "Weather Search Button: Refresh"; `idToLabel` also strips a redundant trailing "Button"/"Btn" word (the role is already announced). A button with real visible TEXT keeps the qualifier prefix (W&B "Pax Level: Randomize" stays distinct).
- **Performance OUTPUT panel is captured ("V1: 155 kt").** The Take-Off / Landing **results** (V1/VR/V2/VRef, Flaps, %N1, RTG, Trim, Sel Temp, Accel Height, Weight) render as `<label class="opt-output" id="opt_takeoff_output_*">155</label>` value cells — which the generic `<label>` text-skip drops, so the entire results panel was invisible after Calculate. A dedicated `collect()` pass scans every visible `.opt-output`, pairs it with its row's `.opt-output-label` name + sibling `.output-unit`, and emits one readable "Name: value unit" static line. An EMPTY (uncalculated) result label collapses to zero size → `isVisible` false → skipped (so the panel only reads once Calculate has run). Live-verified the populated path renders V1/VR/V2/Flaps/Trim/Accel Height; the hidden Take-Off vs Landing sub-panels are gated by `isVisible`.
- **Ground Ops output values + turnaround progress are captured (issue #113, 2026-06).** The Ground Operations app (launched from the tablet HOME screen — `#fs_actions-icon`, NOT the main nav — then the `groundops_automated_groundops` sub-tab) renders its live status as `<label class="groundops_ui_outputlabel">` value-displays — **Turn Time**, **Turn Time Remaining**, Fuel Uplift Remaining, Target Fuel, Uplift — each preceded by a sibling `<label class="groundops_ui_label">` name (+ a `<br>`), with an optional `<span class="output-unit">`. Like the `.opt-output` panel above, the generic `<label>` text-skip dropped ALL of them, so the page showed only the settable Plan-Fuel input + the section headings (the reported bug: "turnaround time … not there"). A dedicated `collect()` pass mirrors the `.opt-output` one: every visible `.groundops_ui_outputlabel` → "Name: value unit" static line (name from the nearest PRECEDING `.groundops_ui_label`, skipping the `<br>`; unit from a following `.output-unit`), empty values skipped so a not-yet-running turn doesn't read blanks. A second pass reads the turnaround **`.progress_bar`** (`.progress_label` "Name<span>- NN%</span>") → "Turnaround: NN%". Both are class-based so they cover EVERY ground-ops sub-page (Maintenance / Connections / Service Vehicles / Doors are button-based and were already captured; their hidden Automated-tab output labels are correctly skipped by `isVisible`). **The fuel outputlabels also carry `pmdg_measurement`** (Target Fuel, the fuel readouts) — which the generic measurement-value path would ALSO capture, emitting an orphan bare "5400 kg" line beside the named "Target Fuel: 5400 kg" (a duplicate masked while the value was the single-char "0", which the glyph-skip drops). So the generic `isMeasure` guard EXCLUDES `.groundops_ui_outputlabel` / `.opt-output` — any label OWNED by a dedicated output pass is treated as non-measurement so only its named line survives. Live-verified on the 777 tablet with a running Short turn (Turn Time 00:55 / Remaining counting down / Turnaround % advancing); jsdom-locked in `tools/pmdg-efb-test/groundops.test.js` (incl. the no-orphan-duplicate guard at a multi-digit Target Fuel).
- **Alert / confirmation cards are auto-ANNOUNCED (2026-06).** EFB alerts render as `#alert_card` (`#alert_card_heading` + `#alert_card_message` + `#alert_card_button` "OK") — they pop up on Save Preferences ("Success — Tablet preferences were updated."), errors, etc. The OK button was already clickable, but the alert TEXT rendered as silent content the screen reader never read. The agent now emits the card as ONE item `type:'alert'` → `_toContract` → `kind:'alert'` + `live:'assertive'`, label = `heading + ': ' + message` (a space is re-inserted after `.!?` before a capital LETTER `[A-Z]` only, since the source `<p>` joins sentences with no space — "updated.Hoppie" → "updated. Hoppie"; a digit is NOT a sentence start, so "12.5" is left intact — `[A-Z0-9]` would have split it to "12. 5"); the card's heading/message are SKIPPED in the main `collect()` loop (the `#alert_card` subtree, except `#alert_card_button`) so there's no duplicate, and the OK button stays a normal clickable button. Host side: `FbwEfbForm.ApplyElements` scans for the `kind=="alert"` element and `AnnounceImmediate`s it (app-side TTS — a dynamically-injected alert isn't reliably spoken by the WebView2 aria-live), once per appearance (`_lastAnnouncedAlert`, cleared when the alert is gone so re-showing speaks again). Fires in BOTH browser + native-list render modes. Live-verified the agent emits the single assertive item + keeps OK; jsdom-locked in `tools/pmdg-efb-test/alertcard.test.js`.
- **Unit toggles render as 2-option COMBOBOXES reading el.checked (UPDATED 2026-06-24 — supersedes the `::after`-caption read).** On the current EFB build the Preferences unit toggles (`efb_preferences_*_unit`, `toggle_switch_text efb_preferences_toggle`) are **TEXTLESS checkboxes**: live-probed via the Coherent debugger, their CSS is literally `content: ""` for BOTH `::after` and `:checked::after` (no unit text at all), AND `Settings` (the bare global — see the `window.Settings`-always-false gotcha below) only COMMITS on "Save Preferences", so it LAGS the live toggle. So the earlier `toggleCaptions(el)[1]` read got an empty caption → fell back to the lagging `Settings` → the displayed unit looked stuck ("doesn't change on toggle"). **The ONLY live signal is `el.checked`.** Fix: `A.UNIT_PAIRS` maps each toggle id → `[uncheckedUnit, checkedUnit]` (verified live by reading each toggle's `checked` + its saved value: length=m/ft, altitude=m/ft, distance=km/nm, weight=kg/lbs, temperature=F/C, pressure=hpa/inhg; **directions differ per toggle** — temp checked=C is metric while length checked=ft is imperial — so the map is per-id, not a universal "checked=metric"). A known unit toggle is reported as a **2-option SELECT** (`item.type='select'`, `options=[unitWord(pair[0]), unitWord(pair[1])]`, `value=unitWord(pair[el.checked?1:0])`) — clearer for a screen reader than a bare "checked" checkbox ("Length unit: feet / meters"). `setValue` on such a select clicks the checkbox to flip to the chosen unit (the `el.type==='checkbox' && UNIT_PAIRS[el.id]` branch). `A.unitWord()` still expands abbreviations (kg→kilograms, nm→nautical miles, hpa→hectopascals, C→Celsius, ft→feet, …). **speed_unit / airspeed_unit are NOT in UNIT_PAIRS** (second value unconfirmed) → they stay checkboxes with the best-effort saved-value read. jsdom-locked (`format.test.js` + `units_tabs.test.js`, unittoggle/prefs fixtures).
- **Active tab / current-page marker (flypad convention) — EFB-WIDE.** `A.activeMarker(el)` appends `" (current page)"` for an active EFB sub-nav page (class `efb_main_menu_button active_button` → "Preferences (current page)") and `" (selected)"` for an active in-page tab. The EFB uses MULTIPLE active-state conventions per area, all covered: `active_button` (nav pages + Performance / Charts / Navdata tabs → "Take Off (selected)") AND a `*_highlighted` suffix (**Ground Operations** sub-tabs, `groundops_menu_button_highlighted` → "Ground Maintenance (selected)" — live-verified 2026-06-21). Detection = `active_button`/`active`/`is-active`/`selected`/`*_highlighted`/aria-selected/aria-current, gated by `isVisible` (hidden pages' persistent active buttons aren't marked). **Excludes `selected-option`** (a dropdown's chosen value, already surfaced as the select's value — not a tab) and the map-layer `_inactive` toggles. `FbwEfbForm` `baseLabel` already strips these suffixes for focus stability. jsdom-locked (tabs + groundtabs fixtures).
- **Duplicate buttons disambiguate by SECTION when the id qualifier is junk.** Two identical buttons (e.g. the two "Import From OFP" on Take-Off, under Airport vs Aircraft) previously read the ugly id qualifier "Import Ofp To Wt: Import From OFP". The disambiguation now strips every label word from the id qualifier; if the remainder is empty / only noise words (to/of/wt/…), it pairs by the nearest enclosing section heading (`_sectionHeading`) → "Airport: Import From OFP" / "Aircraft: Import From OFP". A meaningful id qualifier (W&B "Pax Level") is still kept. All live-verified on the 777 tablet 2026-06-21; jsdom-locked in `tools/pmdg-efb-test` (unittoggle/tabs/dupsection fixtures).

- **Settings (Preferences) + persistent chrome + Dashboard faithfulness pass (2026-06-21, all live-validated on the 777, jsdom-locked).** (1) **Two-column label pairing now covers buttons**: a button claims a same-row label cell to its left (tagged `_isLabel`) so the Preferences "Sign Out"/"Factory Reset" read "Navigraph Authentication: Sign Out" / "Reset to Factory Settings: Factory Reset" instead of orphaning the labels. Row matching is by vertical CENTRE (`_rowTextFor`), so unit toggles centred in taller label rows pair correctly ("AirSpeed Unit" keeps its source casing). VALUE-display items (`_isValue`, e.g. a captured `pmdg_measurement`) are excluded from label pairing so a value never bleeds onto a control. (2) **Persistent chrome (one shared `#efb_menu` + `#StatusBar`, every page) names icon-only buttons by element id, not the FA glyph** (gated to `.efb_main_menu_button` + `^statusbar_`): fixes Paperwork (was "Charts"), Charts (was "Navigation"), the hidden Authenticate, and tablet **Restart** (was "Refresh"); `idToLabel` strips the `statusbar_` prefix and the internal "leaflet" word. Signal + Battery indicators are surfaced as read-only statics WITH their current value (see the status-bar-value note below). (3) **Dashboard**: the whole `.leaflet-container` subtree is skipped (the ~30 map waypoint TOOLTIPS flooded the readout — control buttons + the source select are siblings, so kept); `label.pmdg_measurement` value displays are captured WITH their `.output-unit` so colon-headings merge ("ZFW: 462,330 lb", "ROUTE DIST: 1,183 nm" — fixes a missing value + a "ROUTE DIST: LAX" mis-merge); route-header h3s are labeled by id (Origin/Destination/STD/STA). The map-layer select reads "Map Source" (WORLD/IFR HIGH/IFR LOW/VFR). jsdom fixtures/tests: `tools/pmdg-efb-test/{settings,chrome,dashfull,autocruise,weathersource}.test.js`.

- **Synthetic Preferences controls for config-only settings (no native UI) — table-driven `A.SYN_PREFS`.** The tablet STORES several settings it renders **no control for in v1.7.44** (no DOM node references them); the agent SYNTHESIZES accessible selects for them on the Preferences page (gated on `#efb_preferences` visible + `Settings` available), spliced before the first Preferences action button, at reserved collision-free idx 990001+ (sequential scrape idx never reaches it, so each stays stable across scrapes). `setValue` dispatches idx ≥ 990000 to `A._applySynthetic`. The table (idx, settings key, label, `[[displayText, settingValue]]`):
  - **Weather Source** (`weather_source`, Real World=`REAL-WORLD`/AWC / Sim=`SIM`) → `Settings.updateSetting` (the native setter, which VALIDATES `weather_source` against `window.WeatherSource = {AWC,SIM}` + persists like the unit toggles; the apply maps via the live enum to survive a literal change).
  - **Time Format** (`time_format`, UTC=`utc` / Local=`local`) → `Settings.updateSetting` (no validation on this key).
  - **Map Provider** (`selected_map`, Navigraph=`navigraph` / PMDG=`pmdg`) → drives the LIVE map via `Dashboard.dualMap.switchMap(target)` (the same actuator the Dashboard "Switch Map" button uses) AND `updateSetting` to persist. **Auth-guarded:** switching to Navigraph requires `DualMap.isNGAuthed` — when false, `switchMap` shows its own "Navigraph not authenticated" alert and we RETURN WITHOUT persisting, so the shown value never desyncs from the actual map. (`selected_map` is therefore also reachable via the Dashboard Switch-Map button; the Preferences select is the accessible equivalent.) All three live-verified set→restore on the 777; jsdom-locked in `weathersource.test.js` + `synthprefs.test.js`.
  **⚠️ CRITICAL GOTCHA: `window.Settings` is ALWAYS false on the live PMDG view** — `Settings` is a bare global `const`, never mirrored to `window`. Any code reading it MUST fall back to the bare global (`A.settingsObj()` = `window.Settings || (typeof Settings !== 'undefined' ? Settings : null)`); a `window.Settings`-only check silently no-ops live. **Config audit (live):** the only config-only settings with no native UI are `weather_source`, `time_format`, `selected_map` — ALL three now exposed. Everything else is exposed in Preferences / the Dashboard map controls / the Charts page.
- **Status-bar Signal/Battery show their current VALUE, not just a label** (`A.statusbarValue`): Battery reads the level from the `fa-battery-*` icon class + "charging" from `fa-bolt`/`fa-plug` → "Battery: Full, charging"; Signal reads connectivity from the icon COLOUR (PMDG green `rgb(81,169,72)` → "Connected"; red or a `fa-signal-slash` → "No signal") → "Signal: Connected". (The state is in the icon class / colour, never text — these `div.status-bar-item` indicators have no text content.)
