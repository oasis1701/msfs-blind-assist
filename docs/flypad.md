# FlyByWire Accessible flyPad EFB

> **Read this when:** working on the shared FlyByWire A320/A380 flyPad EFB (Ground, Payload, Fuel, Dashboard, Settings, Doors pages) or its Coherent debugger transport.

### FlyByWire Accessible flyPad EFB (A320 + A380 — ONE shared flyPad)

**Feature:** Screen-reader-accessible Electronic Flight Bag (flyPad) for **both** FlyByWire aircraft. The **same** form, client, and in-page agent serve the A32NX flyPad and the A380X flyPad — only the window title differs. Opened via the shared `HotkeyAction.ShowPMDGEFB` dispatch, branched by `AircraftCode` (`"A320"` and `"FBW_A380"` both call `ShowFBWA380EFBDialog`). It renders **whatever flyPad page is currently open** as a browsable document; there are no curated per-feature tabs.

**Architecture — fundamentally different from the PMDG EFB bridge.** The PMDG/HS787 bridges inject JS via a Community mod package and talk over an HTTP server (`EFBBridgeServer`). The FBW flyPad uses **CDP (Chrome DevTools Protocol) over Coherent GT's devtools port (`:19999`)** with **no mod package and no Community-folder patching** — a generic in-page agent is installed at runtime via `Runtime.evaluate`. `EFBBridgeServer` stays for PMDG/HS787; the FBW flyPad does **not** use it.

```
MSFS FBW flyPad (React / Coherent GT, devtools :19999)   — A32NX or A380X, same "- EFB" view
   window.__MSFSBA_FLYPAD   ← ONE generic in-page agent (installed at runtime)
        ▲ CDP Runtime.evaluate over a persistent WebSocket (/devtools/inspector/{id})
CoherentEFBClient.cs   (.NET ClientWebSocket; IMcduBridge; ~400ms scrape poll → push)
        ▲ fbwa380_efb_connected / fbwa380_efb_elements state pushes
FBWA380EFBForm.cs   (generic WebView2 form; browser mode + native-list fallback)
        ▲ Shift+T (ShowPMDGEFB dispatch; AircraftCode "A320" or "FBW_A380")
```

The A380 reuses this SAME form a third time for the **OANS airport map / BTV**, driven by `CoherentNDClient` (resolves the ND view) instead of `CoherentEFBClient` — see `ShowFBWA380OansDialog`. The form is generic over `IMcduBridge`, with `windowTitle` + `deviceNoun` ("flyPad" / "OANS") parameters.

**Key components:**
- **`Resources/coherent-flypad-agent.js`** — the ONE generic DOM scraper/clicker for both aircraft (Gus Pacleb's A32NX agent, a strict superset of the older A380 one — adds React-onClick detection for handler-less clickable `<div>`s, refuel/boarding action-button state, Ground-service tile detection). **ES5 / Coherent-GT-safe** (`var`, no arrow funcs, `.indexOf()` not `.includes()`, top-level try/catch — a scraper bug returns an error string, never throws into the flyPad). Public global `window.__MSFSBA_FLYPAD`: `ping()`, `scrape()` → `{ok,page,elements[]}`, `clickElement(idx)`, `setValue(idx,text)`, `powerOn()` (sets `L:A32NX_EFB_TURNED_ON=1` — same L:var on both jets — + a synthetic interaction to resume the dormant render loop; idempotent). Each element carries a stamped `idx` (internal attribute `data-fbw-efb-idx`, re-stamped every scrape) plus `kind`/`text`/`value`/`controlType`/`clickable`/`level`/`live`/`disabled`/`options`.
- **`SimConnect/CoherentEFBClient.cs`** — `IMcduBridge` push client over **.NET `ClientWebSocket`** to `ws://127.0.0.1:19999/devtools/inspector/{id}` (the same endpoint `tools/coherent-eval.ps1` uses; it does NOT hit the non-standard-upgrade issue an older manual handshake worked around). Resolves the `- EFB` page from `:19999/pagelist.json` by title every (re)connect (ids shuffle per session); installs the agent once, then `powerOn()`; **polls `scrape()` ~every 400ms** and pushes `fbwa380_efb_connected` / `fbwa380_efb_elements` only when an element-hash (incl. stamped idx) changes. `EnqueueCommand("get_display_elements" | "click_display_element" | "set_element_value")` maps to agent calls. Receive pump correlates responses by CDP message `id`; captured `SynchronizationContext` marshals to the UI thread.
- **`Forms/FBWA380/FBWA380EFBForm.cs`** — the generic **WebView2** form hosting a generated **accessible HTML document** (inline shell, keyed reconcile). Browser mode is primary; a native-list fallback exists if WebView2 fails to init.

**CRITICAL — single-connection constraint.** Coherent GT's debugger accepts only **ONE** devtools connection per page. `CoherentEFBClient` owns the `- EFB` socket while the form is open; it is disposed on the unconditional aircraft-swap cleanup. The `tools/coherent-eval.ps1` harness and the app's client **cannot both be connected at once** — close the flyPad form before driving the page with the tool, and vice-versa. (Verified live on the A32NX via `tools/_probe/drive_tour.ps1`, which tours all 10 tabs and exercises ping/scrape/click/setValue.)

**CRITICAL — the injected flyPad agent can READ SimVars but CANNOT WRITE them (Coherent restriction, live-verified 2026-06).** `SimVar.GetSimVarValue` works from the injected agent (the scrape reads L:vars fine), but `SimVar.SetSimVarValue` from inside an agent function **silently no-ops** — the call returns, no exception, but the L:var does not move. The IDENTICAL `SetSimVarValue` call at the top level of a `Runtime.evaluate` (e.g. a `tools/coherent-eval.ps1` probe) DOES write, and the app's MobiFlight **calc path** (`ExecuteCalculatorCode("{v} (>L:VAR)")`) DOES write — only the agent's own stored-closure call context is blocked from writing. (No load-guard is involved: the agent has none, each `-PreFile` fully replaces `window.__MSFSBA_FLYPAD`; verified `setValue.toString()` contained the new code, `findByIdx` resolved the right stamped element, and a direct write in the same probe eval succeeded while the agent's call did not.) **Consequence:** a settable flyPad control that has **no native `<input>`** — i.e. you'd have to drive it by writing an L:var rather than by `setNativeValue`+Enter or a DOM `click()` — **cannot be wired through the agent.** Controls with a real `<input>` (Payload/Fuel fields) or a real `onClick` button work, because those are DOM interactions, not SimVar writes. **Case study (cabin lighting):** the A380 Quick Controls "Cabin Lighting" is a bare `rc-slider` (no `<input>`/aria) whose value persists `A32NX_CABIN_MANUAL_BRIGHTNESS` via NXDataStore. It scraped + read fine, but every `setValue` path (stamp-based or idx-based) returned "ok" while the L:var stayed put — the agent's `SetSimVarValue` no-op'd. Resolution: the flyPad widget is **suppressed** (`A.suppressCabinLighting` owns the cabin row so it stops surfacing as a misleading "0%Set Cabin Lighting" button), and cabin brightness/auto are exposed as **app-side panel controls** in `FlyByWireA380Definition` (Interior Lighting: `CABIN_BRIGHTNESS_SET` numeric 0-100 + `A32NX_CABIN_USING_AUTOBRIGHTNESS` combo), written via the reliable calc path in `HandleUIVariableSet`. **Rule: if a flyPad control can only be driven by an L:var write (no input, no click), do NOT wire it in the agent — expose it as an app-side panel control instead.**

**CRITICAL — why WebView2 HTML, not native WinForms controls.** The first implementation rendered the scrape as native WinForms controls; in-sim NVDA testing showed **only focusable controls were reachable — headings and static text were invisible** because a WinForms form is not a document and has no browse mode. The renderer hosts a WebView2 accessible HTML document instead, giving NVDA full browse mode (H-key heading nav, arrow-key reading of all text, real fields/links). **The WebView2 shell JS may use modern Chromium syntax** (unlike the ES5 Coherent agent).

**CRITICAL — keyed DOM reconcile (stops NVDA focus jumps).** Do NOT wipe `#root.innerHTML` and rebuild every poll — that steals screen-reader focus. The shell does a **keyed in-place reconcile** (reuse nodes by key, only patch changed text/value/checked, never touch the focused element). The reconcile key is element **CONTENT, NOT the scrape idx** (`kind + controlType + label`), because the agent re-stamps idx from 1 in DOM order on EVERY scrape — so idx is unstable across pages AND across **Ground sub-tabs** (Services/Fuel/Payload/Pushback all report `page` = `"Ground"`). With idx keys, Fuel idx 15 (a button) and Payload idx 15 (an input) collided and cross-patched. Content keys keep a node stable across same-page polls (values patched in place, focus preserved) while a sub-tab/page switch cleanly swaps controls. The live click/setValue idx is carried on each control and patched in place. Because jitter is absorbed here at the reconcile layer, the client can include idx in its push-hash without harming the NVDA browse position. The reference shell `Resources/flypad-shell.html` + its jsdom reconcile tests (`tools/flypad-shell-test/`) are retained as the canonical reconcile spec (they fail on idx keying, pass on content keying); the shipping inline shell mirrors the same logic.

**Sync:** ~400ms auto-poll (client) + F5 manual refresh + delayed re-scrape after a click/setValue. `powerOn()` runs automatically on connect (no wake button). Errors degrade gracefully: no `- EFB` page → "flyPad not detected" silent retry; dormant flyPad → `powerOn()` backoff; CDP timeout/malformed JSON → keep last good render, skip the cycle.

**Dev tool:** `tools/coherent-eval.ps1 -Title "- EFB" -PreFile ...coherent-flypad-agent.js -ExprFile ...` installs + drives the agent against the live page; `tools/_probe/drive_tour.ps1` / `flypad_tour.js` tour every tab. Subject to the single-connection constraint above.

**Ground page tabs + Services accessibility (A320 + A380).** Active-tab indication: the selected nav-rail page reads **"(current page)"** and the selected Ground sub-tab (Services/Fuel/Payload/Pushback) reads **"(selected)"**, both keyed on FBW's standalone `bg-theme-accent` token (inactive carry only `hover:bg-theme-accent`). `orderGroundServices` re-bands the Ground page so the sub-tab strip reads as one group apart from the service tiles (the two-column read order otherwise dropped the strip between the left- and right-column tiles). **Wheel Chocks / Safety Cones are READ-ONLY status, not controls** — FBW renders them as `cursor-pointer` divs with NO click handler whose text colour encodes state (`text-green-500` = placed, `text-gray-500` = not placed; source `A380Services.tsx`/`A320Services.tsx` comment: "only visual information"). `A.groundStatusTile` detects them structurally (language-independent) and surfaces **"Wheel Chocks: placed / not placed"** as a status line; do NOT try to make them clickable (there is no setter — they only become green when ground equipment is placed). The real service tiles (Door/GPU/Stairs/Baggage/Catering/Fuel-truck) keep their existing `(active)`/`(called)` green/amber `serviceState`.

**Ground Payload & Fuel pages — dedicated accessible builders (A320 + A380).** The flyPad Ground→Payload and Ground→Fuel pages are a `<table>` / tank-readout grid wrapped in sighted-only **canvas** visuals (Payload seat map + CG/MAC chart, Fuel wing schematic). A flat positional scrape flattened every table cell into stray fragments (split leading-zero pad spans, lone units, the CG-chart `15%…40%`/`MTOW`/`MLDW`/`MZFW` axis labels) and hid the fuel-target numeric input — unreadable. `coherent-flypad-agent.js` now has two dedicated builders (the `buildFplnLines` pattern), invoked from `enumerate` when the matching sub-tab is active (`A.activeGroundSub(root,'payload'|'fuel')`, detected by the `/ground/<x>` link's standalone `bg-theme-accent` token):
- **`buildPayloadLines`** parses the W&B `<table>` (cols: label / Planned-input / Current-display): re-surfaces each editable `<input>` (stamps `data-fbw-efb-idx` so set/click are unchanged) with `current X, max Y, unit` folded into its accessible name; emits ZFWCG as a read-only line; relabels the per-pax/bag weight inputs (no default-value hint) by pairing them to their weight tooltips in DOM order. Owns (suppresses) every non-actionable section child + the CG-chart card + the table + all absolute `pointer-events-none z-50` tooltips (every label/limit/caption is a tooltip; `findMaxFor` reads the maxes from them first). Leaves the boarding buttons + rate selector to the generic pass.
- **`buildFuelLines`** is scoped to **ROOT, not the tank `h-content-section`** — on the A380 the Total/Trim cards AND the refuel input + rc-slider are absolutely-positioned OUTSIDE the tank section. `fuelVal` matches both layouts (A320 `cur / cap UNIT` single node; A380 `0…0value UNIT` `ValueUnitDisplay` with no capacity) on `textContent` taking the innermost match; `fuelTankName` reads the enclosing `<tr>` first cell / nearest heading / preceding label. The fuel target is identified by the input under the "Refuel" heading and emitted as BOTH an accessible **`<input type="range">` slider AND a number field** sharing the input's idx (both write the same backing input → the in-sim rc-slider follows); slider max = the A320 Total tank's capacity, else the stock `FUEL TOTAL CAPACITY × FUEL WEIGHT PER GALLON` SimVar (the A380 tanks show current-only).
- **`range` control type** is threaded agent → `CoherentEFBClient` (`min/max/step` on `ScrapeElement` → `items.{i}.min|max|step`) → `FbwEfbForm` shell (a `rng` rendered node = `<input type="range">` posting `set` with `controlType:'range'`).
- **`setValue` commits via Enter+blur.** FBW `SimpleInput` commits (clamp + write to the sim) on Enter / blur, NOT on the React `onChange` — so `A.setValue` dispatches a synthetic Enter keydown/keyup + `blur()` after `setNativeValue`. Verified live: setting the fuel target moves `L:A32NX_FUEL_DESIRED` (asynchronously, next frame — a same-tick read still shows the old value).
- **Suppressed everywhere:** the canvas visuals, the rc-slider widget, stringified-boolean leaks (`/^(true|false|undefined|null)$/` — FBW `${cond && t('…')}` template literals render "false" in the Fuel rate-selector tooltips), and the per-page **"Fill … from SimBrief"** controls on both pages (`A.suppressSimbrief` — caption tooltip + the CloudArrowDown icon button, never the value `<input>`; the user imports via the Dashboard). DISABLED FBW buttons carry `pointer-events-none` too (`opacity-20 pointer-events-none`) so tooltip-suppression gates on `absolute` to avoid dropping a disabled Start-deboarding / SimBrief button. Spec + plan: `docs/superpowers/specs/2026-06-03-flypad-ground-payload-fuel-accessibility-design.md`, `docs/superpowers/plans/2026-06-03-flypad-ground-payload-fuel-accessibility.md`.

**Ground page polish — rate selector, time display, door labels, focus.** (1) **Boarding-time display**: FBW renders the boarding duration as a parenthetical CHILD of the label (`<div>Boarding Time<span>(11:00 minutes)</span></div>`), so the generic pass captured only `directText` "Boarding Time" and the parenthetical-skip rule dropped the time. `buildPayloadLines` now emits the element's FULL text ("Boarding Time (11:00 minutes)") and owns it. (Fuel's "Estimated duration … : 0.5" is one text node — already fine.) (2) **Selected rate option** (Instant/Fast/Real): `A.isSelectedSegment` gained a third check — FBW wraps EACH SelectGroup option in its own TooltipWrapper `<div>`, so the SELECTED option's only sibling is its tooltip and the immediate-sibling peer test missed it (symptom: "only Instant showed (selected)"). It now also walks ≤4 ancestors for the SelectGroup container (class `divide-x`) holding ANOTHER `cursor-pointer` option. (3) **"Refuel Time" heading suppressed** in `buildFuelLines` — it labels the rate selector but reads as a redundant, oddly-placed second time line after "Start refueling" (the estimated duration already shows above; the rate options are self-evident, matching the Payload boarding flow which has no separate rate heading). (4) **Door tiles** are grouped together (`orderGroundServices` band 150, gated on `/door/i` + `kind==="button"`) and labeled with the door's REAL name, matched to **MSFSBA's own `_doorDefs` table** (which is also what the read-only door-state announcements speak — so the flyPad label and the spoken name agree). FBW labels several DISTINCT doors with the same tile text ("Door Fwd"/"Door Aft"), and its flyPad enum DIGIT is **unreliable**: e.g. the A380 `Main4Right` controls `INTERACTIVE POINT OPEN:9`, which the FBW MODEL itself (`door_animations.xml`) names **M5R** and the EWD calls **"MAIN 5R"** = MSFSBA's **"Main Door 5 Right"**, NOT "Main 4". So `A.doorIdentity(n)` does NOT trust the digit — it parses the enum NAME from the handler comment (`() => handleButtonClick(N /* Main4Right */)`) and looks it up in the fixed `A.DOOR_NAMES` map (cabinleftdoor→"Forward Left Door", main1left→"Main Door 1 Left", main4right→"Main Door 5 Right", upper1left→"Upper Door 1 Left", …). **Keep `A.DOOR_NAMES` in sync with `FlyByWire{A320,A380}Definition._doorDefs`.** The handler may sit on the stamped node, an ancestor, OR an inner descendant, so it scans n + ≤4 ancestors + the tile subtree. **State-dependence + cache**: FBW sets `onClick=undefined` (and drops the comment) while an attached service (jet bridge/stairs) DISABLES that door, so the identity is only parseable when the door is operable — `doorIdentity` therefore STAMPS the resolved name on the node (`data-fbw-door`, NOT cleared by the per-scrape stale sweep) and reuses that cache when the handler later disappears. **GUARDED**: an unknown enum, a Cargo handler, or a no-comment build returns `""`, and `enumerate` falls back to the robust column-based Left/Right (screen LEFT column = aircraft LEFT side; the `xr`-anchored wrapper holds CabinLeftDoor/Main1Left = interactive point 0) — so the worst case is a side-only label, never a wrong door number. Applied BEFORE the state suffix so the focus-stable key strip still trims `(active)`/`(called)`; "Cargo Door" is matched by `!/cargo/i` and left alone. (5) **Focus jump on activation fixed** (`FbwEfbForm` shell): the WebView2 keyed-reconcile keyed on the FULL label, so a control gaining "(active)"/"(called)"/"(selected)" got a NEW key → its node was destroyed + rebuilt → the screen-reader focus jumped off the control the user just activated. `keyOf` now strips the dynamic state suffixes (`baseLabel`) so the node is reused and patched in place; the visible label still updates. **Verification:** all five are live-verified on BOTH the A320 and A380 (boarding/refuel time, rate-selected state, "Refuel Time" removal, door grouping + precise handler-based labels, tanks/slider). The A380 forward doors read Door Main 1/2 Left + Door Upper 1 Left; the A320's read Door Forward/Aft Left/Right (the A320 handler-based labels are mechanism-verified on the A380 — the A320 enum names map the same way — but re-confirm the exact A320 wording when next loaded). Focus-jump is shell-level (aircraft-agnostic, A320-NVDA-confirmed).
